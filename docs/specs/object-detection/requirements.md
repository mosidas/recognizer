# object-detection — 要件

## はじめに

Recognizer ライブラリ(`docs/api-spec.md` を正とする)の第 2 作業単位。unit `face-detection` が確立した共通基盤(画像入力 3 形式・モデル形式自動判別・前処理・NMS・レターボックス座標復元)を再利用し、公開 API `ObjectDetector` による物体検出(YOLOv5 / YOLOv8 / YOLOv11 形式の ONNX モデル)を提供する。

## スコープ

### 対象(やること)

- 公開 API: `ObjectDetector` / `ObjectDetection`(api-spec 3.5)
- 物体検出モデルの出力形式対応: `F = 4 + クラス数`(YOLOv8/v11)および `F = 5 + クラス数`(YOLOv5)(api-spec 3.2)
- クラス名の解決: `classNames` 指定 / 省略時の COCO 80 クラス名 / `"class_{id}"` フォールバック(api-spec 3.5)
- クラス単位の NMS(api-spec 3.5)
- 画像入力 3 形式のオーバーロードとエラー処理方針の適用(api-spec 3.1, 3.6。face-detection と同一契約)

### 対象外(やらないこと)

- `FaceRecognizer`・顔埋め込み — 理由: unit `face-recognition` の責務(roadmap 参照)
- YOLOv3 形式(3 出力テンソル)— 理由: api-spec 1. でスコープ外
- モデルファイル名によるクラス名リストの自動選択(Open Images 等)— 理由: api-spec 1. でスコープ外
- CPU 以外の実行プロバイダ・ロギング機構 — 理由: api-spec 2., 5. でスコープ外
- 既存 `FaceDetector` の公開契約の変更 — 理由: unit face-detection で凍結済み。共通基盤の internal 拡張は許容する(非回帰は既存テストで担保)

## 前提(未検証の賭け)

unit face-detection で人間承認済みの前提を本 unit にも同一適用する: 閾値の有効範囲は 0.0〜1.0(両端含む、範囲外は `ArgumentException`)/ 返却座標は画像境界内にクリップ / `Dispose` 済みインスタンスへの呼び出しは `ObjectDisposedException`。

本 unit で新規に採用する前提:

- COCO 80 クラス名は英語の標準名(person, bicycle, car, ...。Ultralytics 系 YOLO の学習済みモデルが用いる 80 クラスの順序)とする — 検証方法: requirements 承認ゲートで人間が確認する / 状態: 未検証
- `classNames` 指定時に検出の `ClassId` がリストの範囲外となる場合、例外にせず `"class_{id}"` をクラス名として用いる(モデルの出力形状が動的で構築時にクラス数を確定できない場合があり、構築時の要素数検証を契約にできないため)— 検証方法: requirements 承認ゲートで人間が確認する / 状態: 未検証
- `F = 5 + クラス数`(YOLOv5)の信頼度は objectness とクラススコアの積とする(YOLOv5 の標準解釈)— 検証方法: requirements 承認ゲートで人間が確認する / 状態: 未検証
- 顔検出(F ∈ {5, 20})と物体検出(F = 4 + C / 5 + C)の出力形式判別が両立し得る形状(例: F=20 は「顔」とも「4+16 クラス」とも解釈可能)は、**使用するクラス(FaceDetector / ObjectDetector)側の解釈を正とする**(ObjectDetector は顔専用の F 解釈をしない)。4 + C と 5 + C の両立(同一 F で C が 1 差)の判別規則の詳細は設計フェーズで確定する(api-spec 3.2 に明記)— 検証方法: 設計フェーズで判別規則を確定し design.md に記す / 状態: 未検証

## 用語定義

| 用語 | 定義 |
| ---- | ---- |
| クラス数 C | モデルが識別する物体カテゴリの数。出力の特徴数 F から導かれる(F = 4 + C または 5 + C) |
| COCO 80 クラス | COCO データセット由来の 80 物体カテゴリ(person, bicycle, ... toothbrush) |
| objectness | YOLOv5 形式の出力が持つ「物体が存在する確からしさ」(クラススコアとは独立) |
| クラス単位 NMS | 同一クラスの検出間でのみ重複抑制を行う NMS(異なるクラス同士は抑制しない) |
| 転置形式 / 標準形式 | 出力テンソル形状 `[1, F, N]` / `[1, N, F]`(unit face-detection と同一) |

## 要件

### Requirement 1: 画像入力(3 形式のオーバーロード)

**ユーザーストーリー**: ライブラリ利用者(.NET アプリケーション開発者)として、`FaceDetector` と同じ 3 形式(Mat・ファイルパス・バイト列)で物体検出を呼び出したい。なぜなら API 間で一貫した使い勝手を得たいからである。

**受け入れ基準**:
1.1. システムは、`OpenCvSharp.Mat`(BGR)を `DetectAsync` の画像入力として受け付けなければならない。(常時)
1.2. ファイルパス(`string`)オーバーロードが呼び出されたとき、システムは画像フォーマットを OpenCV の自動判別で読み込み、`Mat` 入力と同一の検出契約で処理しなければならない。(イベント)
1.3. エンコード済み画像バイト列(`ReadOnlyMemory<byte>`)オーバーロードが呼び出されたとき、システムはフォーマットを OpenCV の自動判別でデコードし、`Mat` 入力と同一の検出契約で処理しなければならない。(イベント)
1.4. 画像パスが存在しない・ファイルまたはバイト列が画像としてデコードできない場合、システムは `ArgumentException` を送出しなければならない。(異常系)
1.5. 空の `Mat`(要素数 0)が渡された場合、システムは `ArgumentException` を送出しなければならない。(異常系)
1.6. `null` の `Mat` または `null` の `imagePath` が渡された場合、システムは `ArgumentNullException` を送出しなければならない。(異常系)

### Requirement 2: モデルロードと物体検出形式の自動判別

**ユーザーストーリー**: ライブラリ利用者として、YOLO のバージョン(v5/v8/v11)やテンソルレイアウトを指定せずにモデルパスだけで `ObjectDetector` を構築したい。なぜなら利用者にモデル内部形式の知識を要求したくないからである。

**受け入れ基準**:
2.1. コンストラクタ `ObjectDetector(string modelPath, IReadOnlyList<string>? classNames = null)` が呼び出されたとき、システムは ONNX メタデータから入力レイアウト(NCHW / NHWC)と入力サイズを判別しなければならない。(イベント)
2.2. 入力サイズが動的軸のモデルが与えられる場合、システムは入力サイズ 640x640 を既定として使用しなければならない。(オプション)
2.3. システムは、出力テンソルの転置形式 `[1, F, N]` と標準形式 `[1, N, F]`、および `F = 4 + クラス数`(YOLOv8/v11)と `F = 5 + クラス数`(YOLOv5)を自動判別しなければならない。(常時)
2.4. `F = 5 + クラス数` の形式の場合、システムは各候補の信頼度を objectness と最大クラススコアの積として算出し、`ClassId` を最大クラススコアの添字としなければならない。(オプション)
2.5. `F = 4 + クラス数` の形式の場合、システムは各候補の信頼度を最大クラススコアとし、`ClassId` をそのクラスの添字としなければならない。(オプション)
2.6. `modelPath` のファイルが存在しない場合、システムは `FileNotFoundException` を送出しなければならない。(異常系)
2.7. モデルのロードに失敗した場合、システムは OnnxRuntime の例外をそのまま送出しなければならない。(異常系)
2.8. 入力・出力の形式を判別できない場合(複数の出力テンソルを持つ YOLOv3 系モデルを含む)、システムは `NotSupportedException` を送出しなければならない。(異常系)
2.9. `null` の `modelPath` が渡された場合、システムは `ArgumentNullException` を送出しなければならない。(異常系)

### Requirement 3: クラス名の解決

**ユーザーストーリー**: ライブラリ利用者として、独自モデルにはクラス名リストを渡し、COCO 学習済みモデルには何も渡さずに、人間可読なクラス名を得たい。なぜなら検出結果を UI やログにそのまま表示したいからである。

**受け入れ基準**:
3.1. `classNames` が指定された場合、システムは検出の `ClassId` に対応する名前を `classNames[ClassId]` から解決しなければならない。(オプション)
3.2. `classNames` が省略され、モデル出力からクラス数が 80 と判別できた場合、システムは COCO 80 クラス名を用いなければならない。(オプション)
3.3. `classNames` が省略され、クラス数が 80 以外の場合、システムはクラス名を `"class_{id}"`(例: `class_3`)としなければならない。(オプション)
3.4. `classNames` が指定され、検出の `ClassId` がリストの範囲外の場合、システムは例外を送出せず `"class_{id}"` をクラス名としなければならない。(異常系)

### Requirement 4: 物体検出の実行と結果

**ユーザーストーリー**: ライブラリ利用者として、画像 1 枚から物体のクラス・信頼度・矩形の一覧を取得したい。なぜなら物体の計数・追跡・切り出し等の後段処理の入力にするからである。

**受け入れ基準**:
4.1. `DetectAsync` が呼び出されたとき、システムは信頼度が `confidenceThreshold` 未満の候補を結果から除外しなければならない。(イベント)
4.2. システムは、NMS をクラス単位で適用し(異なるクラスの検出同士は抑制しない)、適用後の検出のみを返さなければならない。(常時)
4.3. システムは、検出結果を信頼度の降順で並べた `IReadOnlyList<ObjectDetection>` として返さなければならない。(常時)
4.4. 検出が 0 件の場合、システムは空リストを返し、例外を送出してはならない。(異常系)
4.5. システムは、`ObjectDetection(int ClassId, string ClassName, float Confidence, RectangleF BBox)` の BBox を入力画像のピクセル座標(左上原点)で返さなければならない。(常時)
4.6. システムは、引数省略時の既定値として `confidenceThreshold = 0.5`・`nmsThreshold = 0.5` を使用しなければならない。(常時)
4.7. `confidenceThreshold` または `nmsThreshold` が 0.0 以上 1.0 以下の範囲外の場合、システムは `ArgumentException` を送出しなければならない。(異常系)

### Requirement 5: 非同期・キャンセル・並行呼び出し・破棄

**ユーザーストーリー**: ライブラリ利用者として、`FaceDetector` と同一の非同期・並行・破棄の契約で `ObjectDetector` を使いたい。なぜなら 2 つの API を同じ運用コードで扱いたいからである。

**受け入れ基準**:
5.1. システムは、すべての非同期メソッドで `CancellationToken`(省略可、既定 `default`)を受け取らなければならない。(常時)
5.2. キャンセルが要求されたとき、システムは `OperationCanceledException`(またはその派生)を送出して処理を中断しなければならない。(イベント)
5.3. システムは、同一インスタンスに対する複数スレッドからの並行 `DetectAsync` 呼び出しのそれぞれに、単独呼び出し時と同一の結果を返さなければならない。(常時)
5.4. `ObjectDetector` は `IDisposable` を実装し、`Dispose` が呼び出されたとき推論セッションを解放しなければならない。(イベント)
5.5. `Dispose` 済みのインスタンスに対して検出が呼び出された場合、システムは `ObjectDisposedException` を送出しなければならない。(異常系)

### Requirement 6: ライブラリ構成(非機能)

**ユーザーストーリー**: ライブラリ利用者として、公開 API が仕様どおりに絞られたまま拡張され、既存の顔検出機能が壊れないライブラリを利用したい。なぜなら既存利用箇所へ影響なくバージョンアップしたいからである。

**受け入れ基準**:
6.1. システムは、本 unit で追加する公開型を `ObjectDetector` / `ObjectDetection` に限定しなければならない(公開型は既存 3 型と合わせ計 5 型)。(常時)
6.2. システムは、本 unit で追加する内部実装(出力パース・クラス名解決等)を `internal` にしなければならない。(常時)
6.3. システムは、コンソール出力・ログ出力をしてはならない。(常時)
6.4. システムは、依存パッケージを既存構成(Microsoft.ML.OnnxRuntime、OpenCvSharp4 + OS 向け runtime)から増やしてはならない。(常時)
6.5. 本 unit の変更後も、既存の全テスト(unit face-detection 分)と新規テストを含む `dotnet build` / `dotnet test` が終了コード 0 で完了しなければならない。(常時)
