# face-recognition — 要件

## はじめに

Recognizer ライブラリ(`docs/api-spec.md` を正とする)の最終作業単位。unit face-detection の `FaceDetector` と共通基盤を再利用し、公開 API `FaceRecognizer` による顔認証(顔埋め込みの抽出と比較)を提供する。ライブラリはコサイン類似度を返すのみで、同一人物か否かの閾値判定は呼び出し側の責務とする(api-spec 1., 3.4)。

## スコープ

### 対象(やること)

- 公開 API: `FaceRecognizer` / `FaceComparisonStatus` / `FaceComparisonResult` / `FaceEmbeddingResult`(api-spec 3.4)
- 顔埋め込みモデルの形式自動判別(入力レイアウト・入力サイズ・1 次元出力ベクトル)(api-spec 3.2)
- 顔領域の切り出し(周辺情報を含む正方形、パディング比率 0.2)(api-spec 3.4)
- `CompareEmbeddings`(コサイン類似度、static)(api-spec 3.4)
- 画像入力 3 形式のオーバーロードとエラー処理方針の適用(api-spec 3.1, 3.6。既存 unit と同一契約)

### 対象外(やらないこと)

- 同一人物か否かの閾値判定 — 理由: api-spec 1. でスコープ外(判定は呼び出し側の責務)
- 1:N 識別(登録済み顔集合との照合)— 理由: api-spec 1. でスコープ外(埋め込み抽出 API の組み合わせで実現可能)
- 顔の角度計算・顔のアライメント(ランドマークによる回転補正)— 理由: 角度計算は api-spec 1. でスコープ外。アライメントは api-spec 3.4 が要求していない(切り出しはパディング付き正方形のみ)
- 既存公開 API(`FaceDetector` / `ObjectDetector` 等 5 型)の変更 — 理由: unit 1・2 で凍結済み。共通基盤の internal 拡張は許容(非回帰は既存テストで担保)
- CPU 以外の実行プロバイダ・ロギング機構 — 理由: api-spec 2., 5. でスコープ外

## 前提(未検証の賭け)

unit 1・2 で人間承認済みの前提を本 unit にも同一適用する: 閾値の有効範囲は 0.0〜1.0(範囲外は `ArgumentException`)/ 返却座標は画像境界内にクリップ / `Dispose` 済みインスタンスへの呼び出しは `ObjectDisposedException`。

本 unit で新規に採用する前提:

- **顔切り出しの具体化**: 検出 BBox の中心を保ち、辺長 = max(幅, 高さ) × (1 + 0.2 × 2) の正方形(各方向へ 0.2 倍ずつ拡張)を切り出す。画像境界をはみ出す部分は境界でクリップする(パディング画素の合成はしない)— 検証方法: requirements 承認ゲートで人間が確認する / 状態: 未検証
- **両画像とも顔未検出の場合**: `FaceComparisonStatus.NoFaceInImage1` を返す(画像 1 を先に評価する)— 検証方法: requirements 承認ゲートで人間が確認する / 状態: 未検証
- **埋め込みベクトル**: モデル出力の生ベクトルを返す(ライブラリ側で L2 正規化しない。正規化はコサイン類似度計算の内部でのみ行う)— 検証方法: requirements 承認ゲートで人間が確認する / 状態: 未検証
- **ゼロベクトルのコサイン類似度**: いずれかの埋め込みがゼロベクトル(ノルム 0)の場合、`CompareEmbeddings` は 0 を返す(数学的に未定義のため安全側の値)— 検証方法: requirements 承認ゲートで人間が確認する / 状態: 未検証
- **埋め込みモデルの前処理詳細**(切り出し画像のリサイズ方式・画素正規化方式・チャネル順)はモデル慣習(ArcFace 系等)の出典を確認して設計フェーズで確定する — 検証方法: 設計フェーズで一次情報を確認し design.md に記す / 状態: 未検証
- **テスト用の埋め込みダミー ONNX** は既存の fixture 方式(生成スクリプト + 生成物コミット)を踏襲して設計フェーズで確定する — 検証方法: 設計フェーズ / 状態: 未検証

## 用語定義

| 用語 | 定義 |
| ---- | ---- |
| 顔埋め込み(embedding) | 顔画像から抽出される固定次元の特徴ベクトル。類似度計算の入力 |
| コサイン類似度 | 2 ベクトルのなす角の余弦。値域 [-1, 1]。1 に近いほど類似 |
| 最高信頼度の顔 | 顔検出結果(信頼度降順)の先頭要素 |
| パディング比率 0.2 | 切り出し正方形を BBox の長辺の 0.2 倍ずつ各方向へ拡張する比率 |
| faceRegion | 利用者が顔検出を省略して指定する顔領域(ピクセル座標の `RectangleF`) |

## 要件

### Requirement 1: 画像入力(3 形式のオーバーロード)

**ユーザーストーリー**: ライブラリ利用者として、既存 API と同じ 3 形式(Mat・ファイルパス・バイト列)で顔比較・埋め込み抽出を呼び出したい。なぜなら API 間で一貫した使い勝手を得たいからである。

**受け入れ基準**:
1.1. システムは、`CompareFacesAsync` と `ExtractEmbeddingAsync` の画像入力として `OpenCvSharp.Mat`(BGR)を受け付けなければならない。(常時)
1.2. ファイルパス(`string`)オーバーロードが呼び出されたとき、システムは OpenCV の自動判別で読み込み、`Mat` 入力と同一の契約で処理しなければならない。(イベント)
1.3. エンコード済みバイト列(`ReadOnlyMemory<byte>`)オーバーロードが呼び出されたとき、システムは OpenCV の自動判別でデコードし、`Mat` 入力と同一の契約で処理しなければならない。(イベント)
1.4. 画像パスが存在しない・ファイルまたはバイト列が画像としてデコードできない場合、システムは `ArgumentException` を送出しなければならない。(異常系)
1.5. 空の `Mat`(要素数 0)が渡された場合、システムは `ArgumentException` を送出しなければならない。(異常系)
1.6. `null` の `Mat` または `null` の `imagePath` が渡された場合、システムは `ArgumentNullException` を送出しなければならない(`CompareFacesAsync` は 2 画像のいずれについても)。(異常系)

### Requirement 2: モデルロードと埋め込みモデルの形式自動判別

**ユーザーストーリー**: ライブラリ利用者として、検出モデルと埋め込みモデルの 2 パスだけで `FaceRecognizer` を構築したい。なぜならモデル内部形式の知識を要求されたくないからである。

**受け入れ基準**:
2.1. コンストラクタ `FaceRecognizer(string detectorModelPath, string embeddingModelPath)` が呼び出されたとき、システムは検出モデルを顔検出の判別規則(unit face-detection と同一)で、埋め込みモデルを ONNX メタデータ(入力レイアウト NCHW/NHWC・入力サイズ)で判別しなければならない。(イベント)
2.2. 埋め込みモデルの入力サイズが動的軸の場合、システムは既定サイズを使用しなければならない(既定値は設計フェーズでモデル慣習の出典に基づき確定する。検出モデルは 640x640)。(オプション)
2.3. システムは、埋め込みモデルの出力を 1 次元ベクトル(バッチ次元を除く要素列)として判別し、その次元数を埋め込み次元としなければならない。(常時)
2.4. いずれかのモデルファイルが存在しない場合、システムは `FileNotFoundException` を送出しなければならない。(異常系)
2.5. いずれかのモデルのロードに失敗した場合、システムは OnnxRuntime の例外をそのまま送出しなければならない。(異常系)
2.6. いずれかのモデルの形式を判別できない場合(埋め込み出力が 1 次元ベクトルと解釈できない場合を含む)、システムは `NotSupportedException` を送出しなければならない。(異常系)
2.7. `null` の `detectorModelPath` または `embeddingModelPath` が渡された場合、システムは `ArgumentNullException` を送出しなければならない。(異常系)

### Requirement 3: 埋め込み抽出(ExtractEmbeddingAsync)

**ユーザーストーリー**: ライブラリ利用者として、画像から顔の埋め込みベクトルを取得したい。なぜなら 1:N 照合などの独自の照合ロジックを呼び出し側で構築したいからである。

**受け入れ基準**:
3.1. `faceRegion` 省略時、システムは顔検出(`detectionThreshold`・`nmsThreshold` 適用)を行い、最高信頼度の顔の領域から埋め込みを抽出しなければならない。(イベント)
3.2. `faceRegion` 省略時の返却 `FaceEmbeddingResult.Face` には、使用した顔の検出結果(`FaceDetection`)を設定しなければならない。(常時)
3.3. `faceRegion` 指定時、システムは顔検出を行わず指定領域から埋め込みを抽出し、`Face` を `null` としなければならない。(オプション)
3.4. 顔領域の切り出しは、対象領域(検出 BBox または指定 `faceRegion`)の中心を保った正方形(辺長 = 長辺 × 1.4、境界クリップ)で行わなければならない(`faceRegion` 指定時も同一の切り出し規則を適用する)。(常時)
3.5. `faceRegion` 省略時に顔が検出されない場合、システムは例外を送出せず `Embedding` と `Face` が共に `null` の `FaceEmbeddingResult` を返さなければならない。(異常系)
3.6. 抽出された `Embedding` は、埋め込みモデルの出力次元数と同じ長さの `float[]` でなければならない。(常時)
3.7. `faceRegion` が空(幅または高さが 0 以下)または画像と交差しない場合、システムは `ArgumentException` を送出しなければならない。(異常系)
3.8. システムは、引数省略時の既定値として `detectionThreshold = 0.7`・`nmsThreshold = 0.5` を使用しなければならない。(常時)
3.9. `detectionThreshold` または `nmsThreshold` が 0.0 以上 1.0 以下の範囲外の場合、システムは `ArgumentException` を送出しなければならない(`faceRegion` 指定で検出を省略する呼び出しでも検証する。既存 unit と同一の常時ガード契約)。(異常系)

### Requirement 4: 顔比較(CompareFacesAsync)

**ユーザーストーリー**: ライブラリ利用者として、2 枚の画像の顔がどの程度似ているかをコサイン類似度で取得したい。なぜなら本人確認の判定閾値を自分のユースケースに合わせて選びたいからである。

**受け入れ基準**:
4.1. `CompareFacesAsync` が呼び出されたとき、システムは各画像で最高信頼度の顔の埋め込みを抽出し、コサイン類似度を `Similarity` に設定した `FaceComparisonResult`(`Status = Success`)を返さなければならない。(イベント)
4.2. 返却の `Face1` / `Face2` には、各画像で使用した顔の検出結果を設定しなければならない。(常時)
4.3. 画像 1 で顔が検出されない場合、システムは例外を送出せず `Status = NoFaceInImage1`・`Similarity = 0`・`Face1 = null` を返さなければならない(両画像とも未検出の場合も `NoFaceInImage1`)。(異常系)
4.4. 画像 2 で顔が検出されない場合(画像 1 は検出)、システムは `Status = NoFaceInImage2`・`Similarity = 0`・`Face2 = null` を返さなければならない。(異常系)
4.5. システムは、同一人物か否かの判定を行ってはならない(返却は類似度のみ)。(常時)
4.6. システムは、引数省略時の既定値として `detectionThreshold = 0.7`・`nmsThreshold = 0.5` を使用しなければならない。(常時)
4.7. `detectionThreshold` または `nmsThreshold` が 0.0 以上 1.0 以下の範囲外の場合、システムは `ArgumentException` を送出しなければならない。(異常系)

### Requirement 5: 埋め込みの比較(CompareEmbeddings)

**ユーザーストーリー**: ライブラリ利用者として、保存済みの埋め込み同士をモデルなしで比較したい。なぜなら登録時に抽出した埋め込みと照合時の埋め込みを軽量に突き合わせたいからである。

**受け入れ基準**:
5.1. `CompareEmbeddings(ReadOnlySpan<float> embedding1, ReadOnlySpan<float> embedding2)` は static メソッドとして、コサイン類似度を返さなければならない。(常時)
5.2. 返却値は -1.0 以上 1.0 以下でなければならない(浮動小数点誤差はクランプする)。(常時)
5.3. 2 つの埋め込みの次元(長さ)が一致しない場合、システムは `ArgumentException` を送出しなければならない。(異常系)
5.4. いずれかの埋め込みがゼロベクトル(ノルム 0)の場合、システムは 0 を返さなければならない。(異常系)
5.5. 同一ベクトル同士の類似度は 1.0(許容誤差内)、逆向きベクトル同士は -1.0(許容誤差内)でなければならない。(常時)

### Requirement 6: 非同期・キャンセル・並行呼び出し・破棄

**ユーザーストーリー**: ライブラリ利用者として、既存 API と同一の非同期・並行・破棄の契約で `FaceRecognizer` を使いたい。なぜなら複数 API を同じ運用コードで扱いたいからである。

**受け入れ基準**:
6.1. システムは、すべての非同期メソッドで `CancellationToken`(省略可、既定 `default`)を受け取らなければならない。(常時)
6.2. キャンセルが要求されたとき、システムは `OperationCanceledException`(またはその派生)を送出して処理を中断しなければならない。(イベント)
6.3. システムは、同一インスタンスに対する複数スレッドからの並行呼び出しのそれぞれに、単独呼び出し時と同一の結果を返さなければならない。(常時)
6.4. `FaceRecognizer` は `IDisposable` を実装し、`Dispose` が呼び出されたとき保持する両推論セッションを解放しなければならない。(イベント)
6.5. `Dispose` 済みのインスタンスに対して非同期メソッドが呼び出された場合、システムは `ObjectDisposedException` を送出しなければならない。(異常系)

### Requirement 7: ライブラリ構成(非機能)

**ユーザーストーリー**: ライブラリ利用者として、公開 API が仕様どおりに絞られたまま完成し、既存機能が壊れないライブラリを利用したい。なぜなら既存利用箇所へ影響なくバージョンアップしたいからである。

**受け入れ基準**:
7.1. システムは、本 unit で追加する公開型を `FaceRecognizer` / `FaceComparisonStatus` / `FaceComparisonResult` / `FaceEmbeddingResult` に限定しなければならない(公開型は既存 5 型と合わせ計 9 型。api-spec の全公開型が揃う)。(常時)
7.2. 公開型のシグネチャ(record 定義・enum 値・メソッド)は api-spec 3.4 と一致しなければならない。(常時)
7.3. システムは、本 unit で追加する内部実装(顔切り出し・埋め込み前処理・コサイン類似度計算等)を `internal` にしなければならない。(常時)
7.4. システムは、コンソール出力・ログ出力をしてはならず、依存パッケージを既存構成から増やしてはならない。(常時)
7.5. 本 unit の変更後も、既存の全テスト(unit 1・2 分)と新規テストを含む `dotnet build` / `dotnet test` が終了コード 0 で完了しなければならない。(常時)
