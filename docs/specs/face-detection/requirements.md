# face-detection — 要件

## はじめに

Recognizer ライブラリ(`docs/api-spec.md` を正とする)の最初の作業単位。プロジェクト骨格(src/Recognizer、tests/Recognizer.Tests)と共通基盤(3 形式の画像入力、ONNX モデル形式の自動判別)を確立し、公開 API `FaceDetector` による顔検出を提供する。後続 unit(object-detection、face-recognition)はこの unit の成果物を再利用する。

## スコープ

### 対象(やること)

- `src/Recognizer`(net10.0 クラスライブラリ)と `tests/Recognizer.Tests`(xUnit)の骨格(api-spec 4.)
- 公開 API: `FaceDetector` / `FaceDetection` / `FaceLandmarks`(api-spec 3.3)
- 画像入力 3 形式のオーバーロード: `Mat` / `string imagePath` / `ReadOnlyMemory<byte>`(api-spec 3.1)
- 顔検出モデルの形式自動判別: 入力レイアウト・入力サイズ・出力形状(api-spec 3.2)
- エラー処理方針の適用(api-spec 3.6)

### 対象外(やらないこと)

- `ObjectDetector` — 理由: unit `object-detection` の責務(roadmap 参照)
- `FaceRecognizer`・顔埋め込み抽出 — 理由: unit `face-recognition` の責務(roadmap 参照)
- 顔の角度計算(roll / pitch / yaw)— 理由: api-spec 1. でスコープ外
- YOLOv3 形式(3 出力テンソル)のモデル対応 — 理由: api-spec 1. でスコープ外
- CPU 以外の実行プロバイダ(GPU 等)— 理由: api-spec 2. で CPU 推論に限定
- ロギング機構・デバッグ用コンソール出力 — 理由: api-spec 1., 5. でスコープ外

## 前提(未検証の賭け)

- 信頼度閾値・NMS 閾値の有効範囲は 0.0 以上 1.0 以下(両端を含む)とし、範囲外は `ArgumentException` とする(api-spec 3.6 の「閾値の範囲外」の具体化)— 検証方法: requirements 承認ゲートで人間が確認する / 状態: 未検証
- 返却する BBox・ランドマーク座標は入力画像の境界内にクリップする(呼び出し側が切り出し等に安全に使えるようにするため)— 検証方法: requirements 承認ゲートで人間が確認する / 状態: 未検証
- `Dispose` 済みインスタンスへの検出呼び出しは `ObjectDisposedException` を送出する(.NET の標準慣習)— 検証方法: requirements 承認ゲートで人間が確認する / 状態: 未検証
- テストで使う ONNX モデルの供給方法(小型ダミー ONNX の生成・実モデルのダウンロード手順等)は設計フェーズで確定する(api-spec 4. に明記)— 検証方法: 設計フェーズで方式を決定し design.md に記す / 状態: 未検証

## 用語定義

| 用語 | 定義 |
| ---- | ---- |
| NMS | Non-Maximum Suppression。重複する検出矩形を IoU に基づいて抑制する後処理 |
| 信頼度(confidence) | モデルが出力する検出の確からしさ。0.0〜1.0 |
| NCHW / NHWC | 入力テンソルのレイアウト(バッチ・チャネル・高さ・幅の並び順) |
| 転置形式 / 標準形式 | 検出モデルの出力テンソル形状。転置形式は `[1, F, N]`、標準形式は `[1, N, F]`(N: 候補数、F: 特徴数) |
| ランドマーク | 顔の 5 特徴点(左目・右目・鼻・口左端・口右端)の座標 |
| 動的軸 | ONNX モデルの入力形状で数値が確定していない軸 |

## 要件

### Requirement 1: 画像入力(3 形式のオーバーロード)

**ユーザーストーリー**: ライブラリ利用者(.NET アプリケーション開発者)として、手元の画像の形態(Mat・ファイルパス・バイト列)のまま顔検出を呼び出したい。なぜなら形式変換のボイラープレートを書かずに済ませたいからである。

**受け入れ基準**:
1.1. システムは、`OpenCvSharp.Mat`(BGR)を `DetectAsync` の画像入力として受け付けなければならない。(常時)
1.2. ファイルパス(`string`)オーバーロードが呼び出されたとき、システムは画像フォーマットを OpenCV の自動判別で読み込み、`Mat` 入力と同一の検出契約で処理しなければならない。(イベント)
1.3. エンコード済み画像バイト列(`ReadOnlyMemory<byte>`)オーバーロードが呼び出されたとき、システムはフォーマットを OpenCV の自動判別でデコードし、`Mat` 入力と同一の検出契約で処理しなければならない。(イベント)
1.4. 画像パスが存在しない・ファイルまたはバイト列が画像としてデコードできない場合、システムは `ArgumentException` を送出しなければならない。(異常系)
1.5. 空の `Mat`(要素数 0)が渡された場合、システムは `ArgumentException` を送出しなければならない。(異常系)
1.6. `null` の `Mat` または `null` の `imagePath` が渡された場合、システムは `ArgumentNullException` を送出しなければならない。(異常系)

### Requirement 2: モデルロードと形式の自動判別

**ユーザーストーリー**: ライブラリ利用者として、モデルの形式(バージョン・テンソルレイアウト)を指定せずにモデルパスだけで `FaceDetector` を構築したい。なぜなら利用者にモデル内部形式の知識を要求したくないからである。

**受け入れ基準**:
2.1. コンストラクタ `FaceDetector(string modelPath)` が呼び出されたとき、システムは ONNX メタデータから入力レイアウト(NCHW / NHWC)と入力サイズを判別しなければならない。(イベント)
2.2. 入力サイズが動的軸のモデルが与えられる場合、システムは入力サイズ 640x640 を既定として使用しなければならない。(オプション)
2.3. システムは、出力テンソルの転置形式 `[1, F, N]` と標準形式 `[1, N, F]`、および `F = 5`(bbox + conf)と `F = 20`(bbox + conf + ランドマーク 5 点 x [x, y, conf])を自動判別しなければならない。(常時)
2.4. `modelPath` のファイルが存在しない場合、システムは `FileNotFoundException` を送出しなければならない。(異常系)
2.5. モデルのロードに失敗した場合、システムは OnnxRuntime の例外をそのまま送出しなければならない。(異常系)
2.6. 入力・出力の形式を判別できない場合、システムは `NotSupportedException` を送出しなければならない。(異常系)
2.7. `null` の `modelPath` が渡された場合、システムは `ArgumentNullException` を送出しなければならない。(異常系)

### Requirement 3: 顔検出の実行と結果

**ユーザーストーリー**: ライブラリ利用者として、画像 1 枚から顔の矩形・信頼度・(モデルが対応する場合)ランドマークを取得したい。なぜなら顔認証や顔領域の切り出し等の後段処理の入力にするからである。

**受け入れ基準**:
3.1. `DetectAsync` が呼び出されたとき、システムは信頼度が `confidenceThreshold` 未満の候補を結果から除外しなければならない。(イベント)
3.2. システムは、`nmsThreshold` を用いた NMS を適用した後の検出のみを返さなければならない。(常時)
3.3. システムは、検出結果を信頼度の降順で並べた `IReadOnlyList<FaceDetection>` として返さなければならない。(常時)
3.4. 検出が 0 件の場合、システムは空リストを返し、例外を送出してはならない。(異常系)
3.5. システムは、BBox を入力画像のピクセル座標(左上原点)の `RectangleF` で返さなければならない。(常時)
3.6. モデルがランドマークを出力する場合(`F = 20`)、システムは 5 点(左目・右目・鼻・口左端・口右端)の `FaceLandmarks` を各検出に含めなければならない。(オプション)
3.7. モデルがランドマークを出力しない場合(`F = 5`)、システムは `FaceDetection.Landmarks` を `null` にしなければならない。(オプション)
3.8. システムは、引数省略時の既定値として `confidenceThreshold = 0.7`・`nmsThreshold = 0.5` を使用しなければならない。(常時)
3.9. `confidenceThreshold` または `nmsThreshold` が 0.0 以上 1.0 以下の範囲外の場合、システムは `ArgumentException` を送出しなければならない。(異常系)

### Requirement 4: 非同期・キャンセル・並行呼び出し・破棄

**ユーザーストーリー**: ライブラリ利用者として、UI やサーバーの応答性を保ったまま検出を実行し、不要になった処理は中断したい。なぜなら CPU 推論は数百 ms 単位の時間を要し、呼び出し側のスレッドを塞ぎたくないからである。

**受け入れ基準**:
4.1. システムは、すべての非同期メソッドで `CancellationToken`(省略可、既定 `default`)を受け取らなければならない。(常時)
4.2. キャンセルが要求されたとき、システムは `OperationCanceledException`(またはその派生)を送出して処理を中断しなければならない。(イベント)
4.3. システムは、同一インスタンスに対する複数スレッドからの並行 `DetectAsync` 呼び出しのそれぞれに、単独呼び出し時と同一の結果を返さなければならない。(常時)
4.4. `FaceDetector` は `IDisposable` を実装し、`Dispose` が呼び出されたとき推論セッションを解放しなければならない。(イベント)
4.5. `Dispose` 済みのインスタンスに対して検出が呼び出された場合、システムは `ObjectDisposedException` を送出しなければならない。(異常系)

### Requirement 5: ライブラリ構成(非機能)

**ユーザーストーリー**: ライブラリ利用者として、公開 API が仕様どおりに絞られた、依存の少ないライブラリを利用したい。なぜなら内部実装への偶発的な依存や依存パッケージの肥大化を避けたいからである。

**受け入れ基準**:
5.1. システムは、名前空間 `Recognizer` に属し、この unit の公開型を `FaceDetector` / `FaceDetection` / `FaceLandmarks` に限定しなければならない。(常時)
5.2. システムは、内部実装(前処理・テンソル変換・NMS・出力パース)を `internal` にしなければならない。(常時)
5.3. システムは、コンソール出力・ログ出力をしてはならない。(常時)
5.4. システムは、依存パッケージを Microsoft.ML.OnnxRuntime と OpenCvSharp4(+ 各 OS 向け runtime パッケージ)に限定しなければならない。(常時)
5.5. システムは、`src/Recognizer` / `tests/Recognizer.Tests` の構成で `dotnet build` と `dotnet test` が終了コード 0 で完了しなければならない。(常時)
