# cli — 要件

## はじめに

recognizer ライブラリ(`src/Recognizer`)は現在ライブラリとしてのみ提供されており、端末から動作を確認したり他のツールと組み合わせて使う手段がない。本 unit は、公開 API(`FaceDetector` / `ObjectDetector` / `FaceRecognizer` と結果型の計 9 型。`docs/api-spec.md` を正とする)を端末から呼び出す CLI 実行ファイル `src/Recognizer.Cli` を新規に追加する。

CLI は推論ロジックを持たず、引数解析・ライブラリ呼び出し・JSON 出力・終了コード決定のみを担う薄い層とする。出力は機械可読な JSON に限り、パイプやスクリプトから利用できるようにする。

## スコープ

### 対象(やること)

- 新規プロジェクト `src/Recognizer.Cli`(既存ライブラリを ProjectReference で参照)と `tests/Recognizer.Cli.Tests` の追加、および `Recognizer.sln` への登録
- 3 コマンド: `detect-face`(顔検出)/ `detect-object`(物体検出)/ `compare-face`(顔類似度)
- JSON 出力(stdout・camelCase・整形なし 1 行)
- エラーの JSON 出力(stderr)と、成功 / 実行時エラー / 使用法エラーを区別する終了コード
- RID 別(`win-x64` / `osx-arm64` / `linux-x64`)の自己完結単一実行ファイルとしての配布手順
- 恒久情報(README ほか)への CLI の使い方の反映

### 対象外(やらないこと)

- 既存ライブラリ(`src/Recognizer`)の公開 API の変更 — 理由: CLI は公開 API の利用者に徹する。API に不足があれば別 unit として起票する
- 同一人物か否かの判定(閾値判定) — 理由: ライブラリ仕様(`docs/api-spec.md` 3.4)がコサイン類似度の返却までを責務とし、判定を呼び出し側に委ねているため。CLI も判定せず類似度を出力する
- 検出結果の画像への描画・可視化・保存 — 理由: 今回の要求仕様に含まれない。JSON 出力のみを提供する
- 標準入力からの画像バイト列受け取り、複数画像の一括処理、ディレクトリ再帰処理 — 理由: 要求仕様は位置引数によるファイルパス入力に限定している(YAGNI)
- JSON 以外の出力形式(人間向け整形テキスト・CSV)と、`--format` 等の出力切替オプション — 理由: 要求仕様が「JSON のみ」を定めている
- ログ出力・進捗表示・詳細度オプション(`--verbose` 等) — 理由: 要求仕様に含まれず、stdout を JSON 専用に保つため
- `win-x64` / `osx-arm64` 実機での実行検証 — 理由: 開発環境は devcontainer(linux/amd64)であり、他プラットフォームの検証は既存 CI の 3 プラットフォームマトリクスに委ねる

## 前提(未検証の賭け)

要求仕様で「設計で確定」とされた項目のうち、要件の判定基準に影響するものについて、本フェーズで採用した前提を記す。設計フェーズでの検証で覆る場合、当該要件に波及する。

- **P1: 閾値(`--confidence` / `--nms` / `--detection-threshold`)の範囲検証は CLI 側で行い、範囲外は使用法エラーとする** — 根拠: ライブラリは閾値範囲外に対して `ArgumentException`(`src/Recognizer/FaceDetector.cs:242` ほか)を送出するが、同じ `ArgumentException` を画像のロード/デコード失敗にも用いている(`src/Recognizer/Internal/ImageDecoder.cs:45`)。例外種別だけでは「使用法エラー(引数不正)」と「実行時エラー(画像ロード失敗)」を区別できないため、閾値検証を CLI が先に行い、ライブラリまで到達した `ArgumentException` は実行時エラーとして扱う。検証方法: 設計フェーズで例外 → code 対応表を確定する際に、この二分が全ケースを覆うことを確認する / 状態: 未検証
- **P2: `--classes` で指定したファイルが存在しない場合は実行時エラーとする** — 根拠: 要求仕様が実行時エラーに挙げる「モデルファイル不在」と同種の外部ファイル不在であり、引数の書式不正(使用法エラー)ではないため。検証方法: 設計フェーズで code 対応表に位置づける / 状態: 未検証
- **P3: `--classes` ファイルの行数がモデルのクラス数と一致しなくても CLI はエラーにしない** — 根拠: ライブラリは `classNames` の範囲外 ID を `class_{id}` にフォールバックする実装であり(`src/Recognizer/ObjectDetector.cs:260`)、CLI が独自に一致を強制するとライブラリ仕様と矛盾するため。検証方法: 設計フェーズでライブラリの該当実装を再確認する / 状態: 未検証
- **P4: テスト用の ONNX モデルは `tests/Recognizer.Tests/Fixtures` の既存ダミー ONNX を再利用する** — 根拠: 要求仕様が再利用を許容しており、CLI テストのために新たなダミーモデル生成手順を持ち込む必要がないため。供給方法(リンク参照・コピー・相対パス解決のいずれか)は設計で確定する / 状態: 未検証
- **P5: CLI の追加は既存 `PublicApiTests` の 2 つの検査に抵触しない** — 根拠: 依存検査は `src/Recognizer/Recognizer.csproj` のみを対象とし(`tests/Recognizer.Tests/PublicApiTests.cs:127`)、コンソール出力の検査は `src/Recognizer` ディレクトリのみを走査する(同 `:107-118`)。いずれも兄弟ディレクトリ `src/Recognizer.Cli` を走査しないため、CLI 固有の依存(`System.CommandLine`)と CLI の stdout 出力は検査範囲外である。検証方法: 実装フェーズで既存テスト 224 件の非回帰を確認する / 状態: 未検証
- **P6: CLI の publish 検証を CI に載せるか否かは設計で判断する** — 根拠: 既存 CI の publish ステップは `dotnet publish src/Recognizer/Recognizer.csproj`(`.github/workflows/ci.yml:39`)とライブラリ限定でハードコードされており、`Recognizer.sln` への登録では CLI の publish は CI に乗らない(`dotnet test` は乗る)。`win-x64` / `osx-arm64` 向け publish の成否を実機で確認するには CI へのステップ追加が要る。検証方法: 設計フェーズで CI 変更の要否を判断し、要件 9.2 の検証手段を確定する / 状態: 未検証

## 用語定義

| 用語 | 定義 |
| --- | --- |
| 実行時エラー | 引数の書式は正しいが、実行中に外部資源(画像・モデル・クラス名ファイル)の読み込みまたは解釈に失敗した状態 |
| 使用法エラー | 引数の書式・組み合わせ・値域が不正で、コマンドを開始できない状態 |
| code | エラー JSON に含める機械可読な文字列識別子。実行時エラーではライブラリの例外種別と発生箇所から、使用法エラーでは CLI が定義する種別から一意に定まる(対応表は設計で確定) |
| 正常な結果 | 検出 0 件(空配列)や `compare-face` の顔未検出を含む、コマンドが最後まで実行できた状態。失敗ではない |
| ダミー ONNX | 推論可能な最小構成の ONNX ファイル。`tests/Recognizer.Tests/Fixtures` に既存のものがある |

## 要件

### Requirement 1: CLI プロジェクトの追加とソリューションへの登録

**ユーザーストーリー**: ライブラリ利用者として、recognizer を端末から実行できる CLI をリポジトリのビルド対象として得たい。なぜなら、ライブラリを組み込むアプリを書かずに動作を確認したいからである。

**受け入れ基準**:
1.1. システムは、`src/Recognizer.Cli` を .NET 10 の実行可能プロジェクトとして持たなければならない。(常時)
1.2. システムは、`src/Recognizer.Cli` から `src/Recognizer` を ProjectReference で参照しなければならない。(常時)
1.3. システムは、`src/Recognizer` の公開 API のみを使用し、`src/Recognizer` の公開 API を変更してはならない。(常時)
1.4. システムは、`tests/Recognizer.Cli.Tests` を xUnit のテストプロジェクトとして持たなければならない。(常時)
1.5. システムは、`Recognizer.Cli` と `Recognizer.Cli.Tests` の両プロジェクトを `Recognizer.sln` に登録しなければならない。(常時)
1.6. `dotnet test` をソリューションに対して実行したとき、システムは既存テスト 224 件をすべて成功させ、かつ CLI テストを実行対象に含めなければならない。(イベント)
1.7. システムは、コマンドラインの解析に `System.CommandLine` を使用しなければならない。(常時)

### Requirement 2: 引数解析と共通オプション

**ユーザーストーリー**: CLI 利用者として、コマンドと引数を誤りなく指定できる手がかりを得たい。なぜなら、閾値やモデルパスの指定ミスを実行前に知りたいからである。

**受け入れ基準**:
2.1. システムは、`detect-face` / `detect-object` / `compare-face` の 3 コマンドを提供しなければならない。(常時)
2.2. システムは、画像をコマンドの位置引数(`detect-face` / `detect-object` は 1 個、`compare-face` は 2 個)としてファイルパスで受け取らなければならない。(常時)
2.3. システムは、`--confidence` の既定値を `detect-face` で 0.7、`detect-object` で 0.5 とし、`--nms` の既定値を 3 コマンドすべてで 0.5、`compare-face` の `--detection-threshold` の既定値を 0.7 としなければならない。(常時)
2.4. 必須オプション(`detect-face` / `detect-object` の `--model`、`compare-face` の `--detector-model` と `--embedding-model`)が指定されなかった場合、システムは使用法エラーとして扱わなければならない。(異常系)
2.5. 位置引数の個数が過不足の場合、または未知のコマンド・未知のオプションが指定された場合、システムは使用法エラーとして扱わなければならない。(異常系)
2.6. 閾値オプションの値が数値として解釈できない場合、または 0.0 以上 1.0 以下の範囲外である場合、システムは使用法エラーとして扱い、ライブラリを呼び出してはならない。(異常系)
2.7. `--help` が指定されたとき、システムはコマンド・オプションの一覧を出力し、終了コード 0 で終了しなければならない。(イベント)

### Requirement 3: detect-face コマンド

**ユーザーストーリー**: CLI 利用者として、画像から検出した顔の位置・信頼度・ランドマークを JSON で得たい。なぜなら、検出結果を後続のスクリプトで処理したいからである。

**受け入れ基準**:
3.1. `detect-face <image> --model <path>` が実行されたとき、システムは `FaceDetector` で顔を検出し、結果を JSON として stdout に出力し、終了コード 0 で終了しなければならない。(イベント)
3.2. システムは、JSON のトップレベルに位置引数で与えられた画像パスを `image`、検出結果の配列を `faces` として含めなければならない。(常時)
3.3. システムは、`faces` の各要素に `bbox`(`x` / `y` / `width` / `height`)、`confidence`、`landmarks` を含めなければならない。(常時)
3.4. ロードしたモデルがランドマークを出力する間、システムは `landmarks` に `leftEye` / `rightEye` / `nose` / `leftMouth` / `rightMouth` の各点(`x` / `y`)を出力しなければならない。(状態)
3.5. ロードしたモデルがランドマークを出力しない間、システムは `landmarks` を `null` として出力しなければならない。(状態)
3.6. 顔が 1 件も検出されなかった場合、システムは `faces` を空配列として出力し、終了コード 0 で終了しなければならない。(状態)
3.7. システムは、`faces` の要素をライブラリの返却順(信頼度降順)のまま出力しなければならない。(常時)

### Requirement 4: detect-object コマンド

**ユーザーストーリー**: CLI 利用者として、画像から検出した物体のクラス・位置・信頼度を JSON で得たい。なぜなら、独自クラスのモデルも含めて検出結果をスクリプトで処理したいからである。

**受け入れ基準**:
4.1. `detect-object <image> --model <path>` が実行されたとき、システムは `ObjectDetector` で物体を検出し、結果を JSON として stdout に出力し、終了コード 0 で終了しなければならない。(イベント)
4.2. システムは、JSON のトップレベルに画像パスを `image`、検出結果の配列を `objects` として含めなければならない。(常時)
4.3. システムは、`objects` の各要素に `classId`、`className`、`confidence`、`bbox`(`x` / `y` / `width` / `height`)を含めなければならない。(常時)
4.4. `--classes <path>` が指定された場合、システムはそのファイルを 1 行 1 クラス名として読み込み、`ObjectDetector` の `classNames` に渡さなければならない。(オプション)
4.5. `--classes` が指定されなかった場合、システムは `ObjectDetector` の既定のクラス名解決(モデル出力が 80 クラスなら COCO 80 クラス名、それ以外は `class_{id}`)に委ねなければならない。(オプション)
4.6. `--classes` で指定されたファイルの行数がモデルのクラス数と一致しない場合、システムはエラーとせず、ライブラリのクラス名解決に委ねなければならない。(状態)
4.7. 物体が 1 件も検出されなかった場合、システムは `objects` を空配列として出力し、終了コード 0 で終了しなければならない。(状態)
4.8. システムは、`objects` の要素をライブラリの返却順(信頼度降順)のまま出力しなければならない。(常時)

### Requirement 5: compare-face コマンド

**ユーザーストーリー**: CLI 利用者として、2 枚の画像の顔のコサイン類似度を JSON で得たい。なぜなら、同一人物かどうかの判定基準を自分で決めたいからである。

**受け入れ基準**:
5.1. `compare-face <image1> <image2> --detector-model <path> --embedding-model <path>` が実行されたとき、システムは `FaceRecognizer.CompareFacesAsync` を呼び出し、結果を JSON として stdout に出力し、終了コード 0 で終了しなければならない。(イベント)
5.2. システムは、JSON に `image1`、`image2`、`status`、`similarity`、`face1`、`face2` を含めなければならない。(常時)
5.3. システムは、`status` を `Success` / `NoFaceInImage1` / `NoFaceInImage2` のいずれかの文字列として出力しなければならない。(常時)
5.4. システムは、同一人物か否かの判定を行わず、コサイン類似度のみを `similarity` として出力しなければならない。(常時)
5.5. `status` が `Success` の間、システムは `face1` / `face2` にそれぞれ使用した顔の `bbox` と `confidence` を出力しなければならない。(状態)
5.6. `status` が `NoFaceInImage1` の間、システムは `similarity` を 0、`face1` と `face2` の両方を `null` として出力し、終了コード 0 で終了しなければならない(ライブラリは画像 1 が未検出の時点で早期返却し、画像 2 を評価しないため。`src/Recognizer/FaceRecognizer.cs:252`)。(状態)
5.7. `status` が `NoFaceInImage2` の間、システムは `similarity` を 0、`face1` に画像 1 で使用した顔、`face2` を `null` として出力し、終了コード 0 で終了しなければならない。(状態)
5.8. システムは、`status` / `similarity` / `face1` / `face2` にライブラリの `FaceComparisonResult` の値をそのまま反映しなければならない(CLI が独自に値を補正しない)。(常時)

### Requirement 6: JSON 出力の共通仕様

**ユーザーストーリー**: スクリプト作成者として、出力形式が一貫した JSON であることを期待したい。なぜなら、パイプで受けて機械的にパースしたいからである。

**受け入れ基準**:
6.1. システムは、成功時の結果をすべて stdout に出力し、stderr には出力してはならない。(常時)
6.2. システムは、JSON のプロパティ名を camelCase で出力しなければならない(`status` の値は列挙子名をそのまま用いる)。(常時)
6.3. システムは、JSON を整形せず 1 行(改行を含まない。末尾の改行 1 個は許容する)で出力しなければならない。(常時)
6.4. システムは、浮動小数点数を実行環境のロケールに依存しない不変(invariant)形式で出力しなければならない。(常時)
6.5. システムは、`image` / `image1` / `image2` に位置引数で与えられた文字列をそのまま出力しなければならない(絶対パスへの正規化をしない)。(常時)

### Requirement 7: エラー出力と終了コード

**ユーザーストーリー**: CLI 利用者として、失敗の理由を機械可読な形で受け取りたい。なぜなら、スクリプトからエラー種別に応じた分岐をしたいからである。

**受け入れ基準**:
7.1. 実行時エラーまたは使用法エラーが発生した場合、システムは `error`(人間可読なメッセージ)と `code`(機械可読な文字列)を持つ JSON を stderr に出力しなければならない。(異常系)
7.2. エラーが発生した場合、システムは stdout に何も出力してはならない。(異常系)
7.3. システムは、終了コードを「成功 = 0」「実行時エラー」「使用法エラー」の 3 種に区別し、実行時エラーと使用法エラーには互いに異なる非 0 の値を割り当てなければならない(具体値は設計で確定する)。(常時)
7.4. 画像ファイルが存在しない、または画像としてデコードできない場合、システムは実行時エラーとして扱わなければならない。(異常系)
7.5. モデルファイルが存在しない、モデルのロードに失敗した、またはモデル形式が非対応の場合、システムは実行時エラーとして扱わなければならない。(異常系)
7.6. `--classes` で指定されたファイルが存在しない場合、システムは実行時エラーとして扱わなければならない。(異常系)
7.7. 実行時エラーが発生した場合、システムは `code` の値をライブラリの例外種別(`FileNotFoundException` / `ArgumentException` / `NotSupportedException` 等)と発生箇所から一意に決定しなければならない(対応表は設計で確定する)。(異常系)
7.8. 使用法エラーが発生した場合、システムは `code` の値を CLI が定義する使用法エラー種別(必須オプション欠落・値の解釈不能・値域外・位置引数の過不足・未知のコマンド/オプション)から一意に決定しなければならない(ライブラリ例外を伴わないため、対応表は設計で確定する)。(異常系)
7.9. 検出 0 件および `compare-face` の顔未検出は、失敗として扱ってはならない(終了コード 0、stderr への出力なし)。(常時)

### Requirement 8: CLI のテスト

**ユーザーストーリー**: 開発者として、CLI の振る舞いが自動テストで守られている状態にしたい。なぜなら、引数解析や JSON 出力の退行を CI で検出したいからである。

**受け入れ基準**:
8.1. システムは、3 コマンドそれぞれについて、正常系の JSON 出力(構造・プロパティ名)と終了コード 0 を検証するテストを持たなければならない。(常時)
8.2. システムは、使用法エラー(必須オプション欠落・閾値範囲外)と実行時エラー(画像不在・モデル不在・非対応モデル形式)について、stderr の JSON(`error` / `code`)と終了コードを検証するテストを持たなければならない。(常時)
8.3. システムは、検出 0 件・顔未検出のケースで終了コード 0 と空配列 / `status` が出力されることを検証するテストを持たなければならない。(常時)
8.4. システムは、テスト用 ONNX モデルとして `tests/Recognizer.Tests/Fixtures` の既存ダミー ONNX を再利用しなければならない(供給方法は設計で確定する)。(常時)
8.5. `dotnet test` が実行されたとき、CLI テストは既存 CI の 3 プラットフォーム(`linux-x64` / `win-x64` / `osx-arm64`)のいずれでも成功しなければならない。(イベント)

### Requirement 9: 配布(自己完結実行ファイル)

**ユーザーストーリー**: CLI 利用者として、.NET ランタイムを別途導入せずに単一の実行ファイルで CLI を使いたい。なぜなら、利用端末に開発環境を構築したくないからである。

**受け入れ基準**:
9.1. `dotnet publish -r <rid>` が実行されたとき、システムは .NET ランタイムを同梱した自己完結の単一実行ファイルを生成しなければならない。(イベント)
9.2. システムは、`win-x64` / `osx-arm64` / `linux-x64` の 3 RID を publish の対象として CLI プロジェクトに設定しなければならない(3 RID すべての publish が成功することの検証手段は、CI へのステップ追加要否とあわせて設計で確定する。前提 P6)。(常時)
9.3. `linux-x64` 向けに publish された実行ファイルは、devcontainer(linux/amd64)上で 3 コマンドを実行でき、要件 3〜7 の出力・終了コードを満たさなければならない。(イベント)

### Requirement 10: 恒久情報への反映

**ユーザーストーリー**: 新規の利用者として、CLI の使い方をリポジトリの恒久情報から知りたい。なぜなら、中間生成物(SDD 成果物)を読まずに使い始めたいからである。

**受け入れ基準**:
10.1. システムは、README に 3 コマンドの構文・オプション・既定値を記載しなければならない。(常時)
10.2. システムは、README に各コマンドの JSON 出力例とエラー JSON の例、終了コードの一覧を記載しなければならない。(常時)
10.3. システムは、README に RID 別の publish 手順を記載しなければならない。(常時)
10.4. システムは、`docs/api-spec.md` §4「リポジトリ構成」に `src/Recognizer.Cli/` と `tests/Recognizer.Cli.Tests/` を追記しなければならない。(常時)
10.5. システムは、`docs/api-spec.md` §2 / §5 の「依存パッケージは 5 件に限る」がライブラリ(`src/Recognizer`)に掛かる制約であり、CLI プロジェクト固有の依存(`System.CommandLine`)を含まないことを明記しなければならない。(常時)
