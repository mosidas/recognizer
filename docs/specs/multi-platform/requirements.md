# multi-platform — 要件

## はじめに

recognizer は現在 linux-x64 でのみ動作する。`src/Recognizer/Recognizer.csproj` が画像処理のネイティブランタイムを `OpenCvSharp4.official.runtime.linux-x64` に固定しており、Windows・macOS では画像処理のネイティブライブラリが解決できないためである。本作業では対応プラットフォームに win-x64 と osx-arm64(Apple Silicon)を追加し、linux-x64 を維持する。

あわせて、この機会に画像処理ライブラリの妥当性(現行 OpenCvSharp の継続可否)を評価軸付きで判断し、対象プラットフォーム上でのテスト自動実行を GitHub Actions で整備する。ライブラリの具体的な採否は設計フェーズの責務であり、本要件は**選定が満たすべき制約と、決定を記録する義務**のみを定める。

## スコープ

### 対象(やること)

- 対応プラットフォームへの win-x64・osx-arm64 の追加(linux-x64 は維持)
- 画像処理ライブラリの選定(継続 / 移行)の評価と決定の記録
- GitHub Actions による対象プラットフォーム上での自動テスト
- 恒久情報(`docs/api-spec.md` / `README.md` / `CLAUDE.md`)への反映
- 依存パッケージ検査テスト(`PublicApiTests`)の許可リスト更新

### 対象外(やらないこと)

- linux-arm64 / win-arm64 / osx-x64 への対応 — 理由: 依頼の対象外。devcontainer が linux/amd64 であり検証手段がなく、CI ランナーも標準では提供されない(win-arm64)ため、需要が生じた時点で別 unit として扱う。これらの RID を指定した `dotnet publish` の成否は検証対象としない(成功しても失敗しても本 unit の合否に影響しない)。
- NuGet パッケージとしての公開・パッケージング(`dotnet pack`・nuspec・バージョニング) — 理由: 依頼に含まれない。ライブラリ利用形態は現時点でプロジェクト参照であり、公開は別途判断する。
- GPU 実行プロバイダの追加 — 理由: api-spec §2 でスコープ外と既定済み。
- ライブラリの機能追加・検出精度の変更 — 理由: 本 unit はプラットフォーム互換性と CI に限定し、振る舞いの非回帰を条件とする。
- CI でのカバレッジ計測・静的解析・リリース自動化 — 理由: 今回の目的はプラットフォーム互換性の実機検証であり、それ以外の CI 機能は YAGNI。

## 前提(未検証の賭け)

- `Microsoft.ML.OnnxRuntime` 1.27.1 は win / osx / linux のネイティブライブラリを 1 パッケージに同梱しており、RID 別パッケージの追加指定なしでマルチプラットフォーム動作する — 検証方法: 設計フェーズで NuGet 公式ページと、`dotnet publish -r <RID>` の出力に各 OS のネイティブライブラリ(`onnxruntime.dll` / `libonnxruntime.dylib` / `libonnxruntime.so`)が配置されることを確認する / 状態: 未検証
- OpenCvSharp の公式ランタイムは osx-arm64 の現行版(4.13 系)を提供しておらず、サードパーティの `Sdcb.OpenCvSharp4.mini.runtime.osx-arm64` が唯一の現行選択肢である — 検証方法: 設計フェーズで NuGet の公式ランタイムパッケージ一覧と最新バージョンを再確認する / 状態: 未検証(依頼者による事前調査あり)
- OpenCvSharp 継続構成(win 公式 + osx-arm64 は Sdcb + linux 公式)は linux(コンテナ)と osx-arm64(実機 Apple Silicon)で全テスト成功する — 出典: 依頼文中の依頼者による事前検証の申告のみ。リポジトリ内にこの構成の痕跡(コミット・csproj・CI 設定)はなく、再現手順も未取得 / 検証方法: 設計フェーズでは再現手順が無いため実証できない。本 unit の CI(Requirement 5)で 3 プラットフォームすべての実機実行を確認して初めて検証済みとする / 状態: **未検証**(依頼者申告)
- GitHub Actions の `macos-14` 以降のランナーは arm64(Apple Silicon)であり、public リポジトリでは macOS ランナーが無料で利用できる — 検証方法: 設計フェーズで GitHub 公式ドキュメント(ランナーイメージ一覧・課金)を確認する / 状態: 未検証
- devcontainer(linux/amd64)では win-x64 / osx-arm64 のネイティブコードを実行できない。したがってこれらのプラットフォームでの動作確認は、ローカルでは `dotnet publish -r <RID>` によるネイティブ資産の配置確認までとし、実行検証は CI に委ねる — 検証方法: 本要件の Requirement 4・5 の受け入れ基準として定義する / 状態: 確定(制約として受け入れる)
- **依頼者に確認せず採用した判断(「お任せ」相当。ゲートで否認があれば差し戻す)**:
  - CI マトリクスに win-x64 を含める(依頼では「追加してよい」= 任意)— 理由: 3 プラットフォームのいずれも devcontainer では実機実行できず、CI が唯一の実機検証手段であるため。win-x64 だけを CI から外すと、その実機動作を確認する手段が本 unit に存在しなくなる。
  - 売上規模等に連動して商用利用に追加ライセンス費用が発生しうるライブラリ(例: Six Labors Split License)は、採用を禁止はしないが、採用する場合は人間の明示承認を要する(Requirement 7.2)— 理由: 本リポジトリは MIT のコアライブラリであり、下流利用者に課金義務を波及させる依存は再利用性を損なう。ただし候補の比較評価自体は依頼で設計フェーズに委ねられているため、要件段階で候補を排除せず、承認事項として扱う。

## 用語定義

| 用語 | 定義 |
| --- | --- |
| RID | .NET の Runtime Identifier。本 unit の対象は `linux-x64` / `win-x64` / `osx-arm64` の 3 つ。 |
| 対象プラットフォーム | 上記 3 RID を指す。 |
| ネイティブ資産 | マネージドアセンブリから P/Invoke されるネイティブ共有ライブラリ(`.so` / `.dll` / `.dylib`)。 |
| 公開 API 契約 | `docs/api-spec.md` §3 が定める公開型・公開メソッドのシグネチャ。現在、公開型は 9 型、画像入力を受ける公開オーバーロードは計 12(3 入力形式 × 4 メソッド)。うち `OpenCvSharp.Mat` を型として露出するのは 4 メソッド(`FaceDetector.DetectAsync` / `ObjectDetector.DetectAsync` / `FaceRecognizer.CompareFacesAsync` / `FaceRecognizer.ExtractEmbeddingAsync` の Mat オーバーロード)。 |
| 非回帰 | 本 unit の変更前に green だった既存テスト 224 件が、変更後も全件 green であること。 |
| 画像処理バックエンド | 次の 6 操作を提供するライブラリ: 画像のデコード / リサイズ / 色変換(BGR→RGB)/ パディング / ROI 切り出し / 画素バッファへの連続アクセス(row-major で取り出し 255 で除して推論テンソルへ充填する。`Internal/Preprocessor.cs` の `GetArray(out Vec3b[])` 相当)。 |

## 要件

### Requirement 1: 対象プラットフォームでのビルドと実行

**ユーザーストーリー**: ライブラリ利用者として、Windows・macOS(Apple Silicon)・Linux のいずれの開発機でも recognizer を参照して動かしたい。なぜなら、チームの開発機と CI が単一 OS に固定されていないからである。

**受け入れ基準**:

1.1. システムは、`linux-x64` / `win-x64` / `osx-arm64` の 3 つの RID すべてに対して、画像処理バックエンドのネイティブ資産を解決できなければならない。(常時)
1.2. `dotnet publish -r <RID>` が対象プラットフォームの RID に対して実行されたとき、システムは publish 出力ディレクトリに当該 RID 向けの画像処理バックエンドのネイティブ資産と ONNX Runtime のネイティブ資産を配置しなければならない。(イベント)
1.3. RID を指定せずに `dotnet build` / `dotnet test` が実行された場合、システムは実行中のホストプラットフォーム(devcontainer では linux-x64)で従来どおりビルド・テストを終了コード 0 で完了しなければならない。(イベント)
1.4. システムは、対象プラットフォームのうち特定の 1 つでのみ有効なネイティブランタイムパッケージを、他のプラットフォーム向けの publish 出力に含めてはならない。(常時)

### Requirement 2: 画像処理ライブラリの選定と決定の記録

**ユーザーストーリー**: メンテナとして、画像処理ライブラリを継続するか移行するかを根拠付きで決めたい。なぜなら、この選択が公開 API・ライセンス・サプライチェーンリスクに長期的な影響を与えるからである。

**受け入れ基準**:

2.1. システム(設計成果物)は、画像処理バックエンドの候補を比較評価し、採否とその根拠を `docs/specs/multi-platform/design.md` に記録しなければならない。(常時)
2.2. システム(設計成果物)は、比較評価において次の 5 評価軸をすべて明示的に扱わなければならない: 公開 API 契約への影響 / ライセンス / 供給元の信頼性(公式かサードパーティか) / 対象 3 プラットフォームでのネイティブ資産の入手可否 / 用語定義に列挙した 6 操作(デコード・リサイズ・色変換・パディング・ROI 切り出し・画素バッファへの連続アクセス)の充足。(常時)
2.3. 選定結果がサードパーティ(非公式)提供のパッケージに依存する場合、システム(設計成果物)は、そのパッケージを採用する理由・供給停止時の影響・代替への撤退条件を記録しなければならない。(異常系)
2.4. 選定結果が現行の画像処理バックエンド(OpenCvSharp)の置換を伴う場合、システムは実装フェーズに進まず、作業単位の再分割案を提示して人間の承認を求めなければならない(公開 API 契約の破壊的変更を伴うか否かを問わない)。(異常系)
2.5. 選定において、対象 3 プラットフォームのいずれかでネイティブ資産または実行可能な代替が存在しない候補は、採用してはならない。(異常系)

### Requirement 3: 公開 API 契約の維持

**ユーザーストーリー**: ライブラリ利用者として、マルチプラットフォーム対応によって既存の呼び出しコードが壊れないことを保証してほしい。なぜなら、プラットフォーム追加は利用者から見て機能追加であり、破壊的変更を伴う理由がないからである。

**受け入れ基準**:

3.1. システムは、`docs/api-spec.md` §3 が定める公開型を 9 型に保ち、増減させてはならない。(常時)
3.2. システムは、既存の公開メソッドのシグネチャ(名前・引数の型と順序・戻り値型)を変更してはならない。(常時)
3.3. システムは、内部実装型(前処理・NMS・出力パース等)を公開してはならない。(常時)
3.4. 画像処理バックエンドの変更により公開 API 契約の維持が不可能と判明した場合、システムは Requirement 2.4 の手順に従うものとし、要件・ルーティングの再承認を経ずに実装してはならない。(異常系)

### Requirement 4: 既存テストの非回帰と依存検査の整合

**ユーザーストーリー**: メンテナとして、プラットフォーム追加が既存の振る舞いを壊していないことをテストで確認したい。なぜなら、本 unit は互換性の作業であり、振る舞いの変更を意図していないからである。

**受け入れ基準**:

4.1. 変更後に `dotnet test` が実行されたとき、システムは既存テスト 224 件を全件 green で完了しなければならない(Failed 0 / Skipped 0)。(イベント)
4.2. `src/Recognizer/Recognizer.csproj` の `PackageReference` が変更されたとき、システムは `PublicApiTests` の依存パッケージ許可リストを同一の集合へ更新し、テストを green に保たなければならない。(イベント)
4.3. システムは、依存パッケージ許可リストと `Recognizer.csproj` の `PackageReference` 集合が厳密一致することを、テストで検証しなければならない。(常時)
4.4. 画像処理バックエンドが置換された場合、システムは用語定義に列挙した 6 操作それぞれについて、振る舞いが置換前と同等であることを検証するテストを追加しなければならない。(異常系)
4.5. devcontainer 上で win-x64 / osx-arm64 のネイティブコードを実行できないため、ローカル検証は `dotnet publish -r <RID>` によるネイティブ資産の配置確認までとし、実行検証は Requirement 5 の CI に委ねなければならない。(常時)

### Requirement 5: CI による対象プラットフォームでの自動テスト

**ユーザーストーリー**: メンテナとして、3 つの対象プラットフォームすべてで実機テストが自動実行されてほしい。なぜなら、devcontainer では win-x64 / osx-arm64 の実行検証ができず、CI が唯一の検証手段だからである。

**受け入れ基準**:

5.1. システムは、GitHub Actions のワークフロー定義ファイルを `.github/workflows/` 配下に持たなければならない。(常時)
5.2. main ブランチへの push または main ブランチ宛の pull request が発生したとき、システムはワークフローを起動しなければならない。(イベント)
5.3. システムは、`ubuntu-latest`(linux-x64)・`windows-latest`(win-x64)・`macos-14` 以降の Apple Silicon ランナー(osx-arm64)の 3 ジョブをマトリクスで実行しなければならない。(常時)
5.4. 各ジョブが実行されたとき、システムは .NET 10 SDK をセットアップし、`dotnet build` と `dotnet test` を実行しなければならない。(イベント)
5.5. いずれかのジョブでビルドまたはテストが失敗した場合、システムはワークフロー全体を失敗として報告しなければならない。(異常系)
5.6. 1 つのジョブが失敗した場合でも、システムは他のジョブを完走させ、全プラットフォームの結果を得られるようにしなければならない。(異常系)
5.7. 本リポジトリの devcontainer では GitHub Actions を実行できないため、システムはワークフロー YAML の構文妥当性と、実行コマンドがローカルの `dotnet build` / `dotnet test` と整合することをローカル検証の範囲としなければならない。(常時)

### Requirement 6: 恒久情報の更新

**ユーザーストーリー**: 次の読み手(将来のメンテナ・ライブラリ利用者)として、対応プラットフォームと依存パッケージの実態をリポジトリの恒久情報から知りたい。なぜなら、中間生成物(`docs/specs/`)は unit 完了で凍結され参照専用になるからである。

**受け入れ基準**:

6.1. システムは、`docs/api-spec.md` §2「対象環境」の動作環境を、対応する RID(`linux-x64` / `win-x64` / `osx-arm64`)を明示した記述へ更新しなければならない。(常時)
6.2. システムは、`docs/api-spec.md` §2「画像処理」と §5「非機能要件」の依存パッケージ記述を、選定結果(実際の `PackageReference`)と一致させなければならない。(常時)
6.3. システムは、`README.md` に対応プラットフォームと CI での検証方法を記載しなければならない。(常時)
6.4. システムは、`CLAUDE.md` の devcontainer に関する注意事項を、選定結果を踏まえた実態(linux-arm64 非対応の理由が有効か否かを含む)へ更新しなければならない。(常時)
6.5. 凍結済みの `docs/specs/face-detection/` / `docs/specs/object-detection/` / `docs/specs/face-recognition/` の中間生成物を、システムは変更してはならない。(常時)

## 非機能要件

### Requirement 7: 依存パッケージの健全性

**ユーザーストーリー**: ライブラリ利用者として、依存パッケージが供給停止・ライセンス上の制約で使えなくなるリスクを避けたい。なぜなら、コアライブラリの依存は下流の全利用者に波及するからである。

**受け入れ基準**:

7.1. システムは、依存パッケージを ONNX Runtime と画像処理バックエンド(+ その RID 別ランタイムパッケージ)に限定しなければならない。(常時)
7.2. 売上規模等に連動して商用利用に追加ライセンス費用が発生しうるライブラリを選定結果に含める場合、システムは実装フェーズに進まず、そのライセンス条件を提示して人間の明示承認を求めなければならない。(異常系)
7.3. システムは、依存パッケージ(RID 別ランタイムパッケージを含む)のライセンス名を設計成果物に記録しなければならない。(常時)
7.4. 画像処理バックエンドが置換された場合、システムは devcontainer 上で同一入力(640×640 の画像 1 枚)に対する前処理(デコードから推論テンソル充填まで)を 100 回実行した所要時間の中央値を置換前後で比較し、置換後が置換前の 2 倍を超えたときは、その計測値と採用可否の再判断を設計成果物に記録しなければならない。(異常系)
