# dev-requirements — EARS 要件の生成

依頼内容をテスト可能な要件に構造化し、`<workdir>/requirements.md` を生成するサブ部品。出力は実装までの中間生成物であり、実装完了で凍結される(受け入れ基準はテストコードへ引き継がれる)。

## 1. 契約

- **workdir**: 呼び出し時の引数で指定する。既定は `docs/dev/`(composition は上書きする)。
- **入力**: 依頼文(引数または対話)。既存の `<workdir>/requirements.md` があれば更新モード(全置換せず差分更新)。
- **出力**: `<workdir>/requirements.md`(EARS 受け入れ基準・要件 ID 連番)。
- **port**: 差し替え port `hearing-questions`(name 参照。ヒアリング質問セット)。知識 port を含む選択はポートマッピング(`../dev-core/references/ports.md`)に従う(inject に dev-requirements を含むもの。例: 原則との整合、移行の規律)。

## 2. 参照(必読)

- EARS 記法: `./references/ears-format.md`
- 記法規約(要件 ID・マーカー・曖昧語禁止): `../dev-core/references/notation.md`
- Git 運用規約: `../dev-core/references/git-convention.md`
- 文書ゲート: `../dev-core/templates/doc-gate-prompt.md`
- テンプレート: `./templates/requirements-template.md`

## 3. ステップ

### Step 1: コンテキスト読込

- 依頼文と、既存の `<workdir>/requirements.md`(あれば)を読む。
- 知識 port をポートマッピング(`../dev-core/references/ports.md`。走査は `ports.py --skill dev-requirements`)に従って読み、原則・規律と矛盾しない要件にする。差し替え port `hearing-questions` の有無もこの走査結果で確認する。
- **既存コントラクトの確認(既存コードベース拡張時)**: エラー型とステータスコード・API クライアントの分岐・採用済みテスト基盤など、既存実装の実シグネチャを確認し、推測で要件化しない。大量の読み込みが要る場合は dev-explorer に隔離してダイジェストのみ受け取る。
- workdir が凍結済み(state.json が完了状態)の場合は再生成せず停止し、新しい作業単位(別 workdir)を案内する。

### Step 2: ヒアリング(質問駆動の要求確定)

要件化の前に、曖昧な点を `AskUserQuestion` で確認する(1〜3 問ずつ)。

- **既定(簡易)**: 推測禁止項目 = 利用者・ロール / 準拠レベル(セキュリティ・規制) / 技術スタック・インフラ制約 / ドメインルールのうち、依頼文から確定できないものだけを質問する。加えて、機能の境界(含む/含まない)と非機能要件の有無を確認する。
- **GUI を持つ作業の必須確認**: 事前に指定された視覚設計の正本(デザイントークン定義・共通レイアウト仕様・デザインシステム・UI モック)の有無と所在を必ず確認する。正本が存在するのに要件へ乗らないと、下流(設計のトレース・実装・レビュー)の全検証から外れる。
- **認証を持つ作業の必須確認**: 認証方式(セッション Cookie・トークン等)、初期ユーザー(seed)の供給方法、セッション永続戦略(リロード・再訪問後の復元経路)、セッション失効時の想定挙動のうち、依頼文から確定できないものを必ず質問する。
- **port 差し替え時**: `name: hearing-questions` の port があれば、その質問セット(顧客ヒアリング等の深い体系)に従う。
- 推測で確定できない点はすべて質問し切り、残さない。ユーザーが「お任せ」と答えた項目は、選んだ案と理由を要件の「前提」に明記する。

### Step 3: EARS 要件の生成

`./templates/requirements-template.md` に従って requirements.md を作成する。

- 各要件は数値 ID(`Requirement 1:`)とユーザーストーリー(〜として、〜したい。なぜなら〜)を持つ。
- 受け入れ基準は EARS の 5 パターン(常時 / イベント / 状態 / 異常系 / オプション)で記述し、`1.1`, `1.2` と番号付けする。
- 異常系・エッジケースを必ず検討する。セキュリティ関連機能では、各ユースケースの誤用シナリオ(abuse case: 不正入力・権限逸脱・なりすまし等)を洗い出し、異常系受け入れ基準の発生源にする(設計の脅威モデルと往復する)。
- 認証を持つ機能では、攻撃耐性(abuse case)とは別に、正常系の成立と永続性を受け入れ基準に必ず含める: ログインできること(イベント)、ログイン後にリロード・再訪問してもセッションが維持されること(状態)、セッション失効時の挙動(異常系)。「破られない」と「動く」は別の基準であり、片方で他方を代替しない。
- GUI を持つ機能では、共通レイアウト(ヘッダ等)が全画面に適用されること、色・書体・余白が指定デザイントークンに準拠することを受け入れ基準化する(視覚設計の正本があるときは所在を参照で示す)。
- データ削除のカスケードで新たに生じる状態(null 化した参照・親を失ったレコード等)について、その状態で取り得る操作を状態系/異常系の受け入れ基準として定義する。フレームワーク既定の挙動に委ねて未定義のまま残さない。
- 曖昧語を使わず定量化する(`../dev-core/references/notation.md` §4)。

### Step 4: 内蔵レビューゲート

`../dev-core/templates/doc-gate-prompt.md` で dev-reviewer を独立文脈で 1 体起動し、文書ゲート系 3 観点(correctness / testability / completeness)で判定させる。種別固有チェックとして以下を渡す。

- 網羅性: 依頼・ヒアリングで確定したスコープを漏れなくカバーしているか。任意とされた項目を勝手に必須化していないか。
- EARS 準拠: 各受け入れ基準が EARS 構文と数値 ID 規則に従うか。
- スコープ境界: やらないことが理由付きで明示されているか。
- 前提の明示: 未検証の前提が「前提」セクションに検証方法付きで書かれているか。
- 隣接期待: ユーザーが暗黙に期待しそうな要件の取りこぼしがないか。
- 機能横断の必須基準: 認証を持つ機能に、ログイン正常系・セッション永続(リロード耐性)・失効時挙動の受け入れ基準があるか。GUI を持つ機能に、共通レイアウトの適用・指定デザイントークン準拠の受け入れ基準があるか。
- 参照の実在: 引用した既存コード・文書の参照が実在するか(捏造禁止)。

`REJECTED` なら指摘を反映して**最大 2 回**自己修復する。`QUESTIONS` があれば `AskUserQuestion` でユーザーに確認して解消する(推測で埋めない)。

### Step 5: 保存とコミット

- `<workdir>/requirements.md` を保存する。
- `check.py --workdir <workdir>` で機械検査する(要件 ID の連番・残存マーカー・曖昧語。state.json がある場合は `--def` も付けて状態検査。詳細は `../dev-core/references/static-check.md`)。warning は内容を確認して Step 4 の自己修復に戻す。
- `../dev-core/references/git-convention.md` に従い選択的にステージして commit & push する(例: `docs(<unit>): EARS 要件を生成`)。

### Step 6: 停止

- requirements.md を提示し、ユーザーのレビューを待って停止する。**状態遷移(承認)はこの部品では行わない**(composition またはユーザーが状態機械を操作する)。
- 次の部品として dev-design(設計)を案内する。

## 4. 注意

- 要件には実装詳細(具体的な技術・ライブラリ選定)を書かない。それは設計(dev-design)の責務。
- 要件番号は design / tasks からトレースされる。安易な番号変更を避ける。
- 更新モードでは既存の要件 ID を保持し、追加は末尾連番、削除は欠番として理由を残す。
