# Git 運用規約(commit / push / PR / CI)

本スキル群の各スキルが **ドキュメントやソースコードを更新したら、その時点で commit し push する** ための共通規約。あわせて、作業ブランチの PR 作成と CI 追従の手順(8.)を定める。各 SKILL.md から参照される。

## 1. 目的(なぜ commit / push するか)

- **詳細なプロセスのログを残すため**: 各工程(要求整理・設計・タスク分解・実装等)の成果物をコミット単位で履歴化し、誰がいつ何を生成・変更したかを追跡可能にする。
- **AI による意図しない環境破壊のバックアップのため**: 各ステップで履歴を確定しておけば、誤った変更が起きても直前の状態へ復帰できる。push によりリモートにも退避する。

## 2. 適用範囲

- **文書を生成・更新する部品**(要求整理・設計・タスク分解等): 成果物の保存後に commit & push する。
- **コードを変更する部品**(実装): 各タスクの実装が完了し検証を通った時点で commit & push する。
- **read-only の部品**(整合性検査等): ファイルを変更しないため commit しない。

## 3. ブランチ運用

ブランチ運用(ブランチの命名・作成タイミング・統合方法)は composition(ワークフロー)側が定める。本リファレンスの commit / push は現在のブランチに対して行う。

## 4. タイミング

- **保存時すぐにコミットする(承認前でもよい)**。承認ゲートの前に成果物を履歴化することで、プロセスログとバックアップの目的を満たす。
- 成果物ファイルと、同時に更新する状態ファイル(state.json 等)は **同一コミット** にまとめる。
- 承認後に状態ファイルの承認フラグ・状態だけを更新した場合は、メタデータのみの小さなコミットとして別途コミットする(type は `chore`)。

## 5. コミットメッセージ形式(Conventional Commits 1.0.0)

形式: `<type>(<scope>): <description>`

出典: Conventional Commits 1.0.0 https://www.conventionalcommits.org/ja/v1.0.0/

- **description**: 日本語で簡潔に、何をしたかを命令形〜要約で記す。
- **本文**: 空行を挟み、**変更理由(Why)** を記す(「コミットログには Why」。[durable-info.md](./durable-info.md))。自明な変更では省略してよいが、判断・トレードオフを含む変更では必須とする。

### 5.1. type(用途別)

| type       | 用途                                                       | 主に使う部品             |
| ---------- | ---------------------------------------------------------- | ------------------------ |
| `docs`     | ドキュメント(要件・設計・タスク・レポートなど)の追加・更新 | 文書を生成・更新する部品 |
| `feat`     | 新機能のコード追加                                         | コードを変更する部品     |
| `fix`      | バグ修正のコード変更                                       | コードを変更する部品     |
| `test`     | テストの追加・修正                                         | コードを変更する部品     |
| `refactor` | 振る舞いを変えないコード整理                               | コードを変更する部品     |
| `ci`       | CI 設定・パイプライン(ワークフロー定義・品質ゲート)の変更  | コードを変更する部品     |
| `chore`    | ビルド・設定・メタデータ(状態ファイルの承認更新など)の変更 | 各部品                   |

### 5.2. scope

- **作業単位の成果物**: 対象の作業単位名(issue・feature 等の識別子。workdir のディレクトリ名がある場合はそれに合わせる。例: `docs(user-auth): 受け入れ基準を生成`)。
- **コードの変更**: 変更対象のモジュール・コンポーネント名、または作業単位名(例: `feat(api): エンドポイントを追加`)。
- scope が自明でない・横断的な場合は省略してよい(例: `docs: ...`)。

### 5.3. 例

```
docs(user-auth): 仕様(契約と受け入れ基準)を生成(spec.md)
docs(payment): 設計の曖昧点を質問駆動で解消
feat(user-auth): ログイン API のハンドラを実装
test(user-auth): トークン失効の異常系テストを追加
refactor(payment): 決済ハンドラの重複を集約
chore(user-auth): 承認に伴い state.json を更新
```

## 6. 安全制約(厳守)

- **破壊的操作の禁止**: `git checkout .`、`git checkout <commit> -- <path>`、`git reset --hard`、`git clean -f`、`git branch -D`、`rm -rf`(再帰強制削除)、未コミット変更を失う操作を行わない。
- **選択的ステージング**: 今回の成果物に対応するファイルだけを明示的に `git add <file>` する。`git add -A` / `git add .` は使わない。無関係な変更を巻き込まない。
- push はデフォルトブランチや共有ブランチへの強制 push(`--force`)を行わない。

## 7. 実行手順

1. **リポジトリ確認**: `git rev-parse --is-inside-work-tree` で git 管理下か確認する。
   - **git 管理下でない場合**: commit / push を **スキップして続行**する。その旨を一度だけユーザーに通知し、プロセスログ・バックアップが欲しい場合は `git init` を提案する(自動では init しない)。
   - **init する場合の規約**: `git init` を行う場合は既定ブランチを `main` とする(`master` を避ける)。実行環境(devContainer 等)が初期化を自動化している場合はそれに従ってよいが、そのような環境を前提にしない。
2. **ステージング**: 成果物ファイル(および同時更新する状態ファイル)を `git add <file>` で個別に追加する。
3. **コミット**: 上記の Conventional Commits 形式で `git commit` する。
4. **リモート確認と push**: `git remote` でリモートの有無を確認する。
   - **リモートがある場合**: 現在のブランチを push する(`git push`。未設定なら `git push -u <remote> <branch>`)。
   - **リモートが無い場合**: ローカルコミットのみ実施し、push を **スキップ**してその旨を通知する。
5. **push 失敗時**: 認証・非 fast-forward 等で push が失敗しても **ワークフローは止めない**。ローカルコミットは保持し、失敗内容と推奨対応(`git pull --rebase` 等)をユーザーに報告する。

## 8. PR 作成と CI 追従

作業ブランチの成果を main 等へ統合する前の PR 作成と、PR 上で走る CI(自動テスト・ビルド・リント)の追従の手順・規律。実行のタイミング・要否・失敗修正の実施方法は composition 側が定める(部品が本節を単独で実行しない)。**PR のマージは行わない**(人間に委ねる)。

### 8.1. 適用条件

- git 管理下・リモートあり・`gh` が利用可能(`gh auth status` で認証済み)であること。満たさなければ本節をスキップし、その旨を一度だけ報告して続行する(ワークフローは止めない)。
- リポジトリに CI が定義されていない場合(PR にチェックが 1 つも付かない場合)は、PR 作成のみ行い CI 追従をスキップする(その旨を報告する)。

### 8.2. PR 作成

- push 済みの作業ブランチから、composition が定める統合先(通常 main)へ PR を作成する(`gh pr create --base <統合先>`)。
- タイトルはコミットメッセージ形式(5.)に準じ、本文には変更の要約・検証結果(テスト・最終検証の結果)・レビューで見てほしい点を日本語で記す。
- 同一ブランチの open な PR が既にあれば(`gh pr view` で確認)、新規作成せずそれを対象に CI 追従へ進む。

### 8.3. CI 追従(グリーンまで)

1. `gh pr checks <PR> --watch` で CI の完了を待つ(watch が使えない環境ではポーリングで代替する)。
2. 全チェック成功 → PR の URL と CI 結果を報告して本節を完了する。
3. 失敗があれば、`gh run view <run-id> --log-failed` で失敗ログを取得し、原因を特定して修正する(修正の実施規律は composition が定める。例: dev-implement の軽量タスク定義)。修正の commit & push で CI が再実行されるため、1. へ戻る。
4. **有界リトライ**: 修正ラウンドは **1 PR あたり最大 3 回**。超えたら停止し、失敗しているチェック・試行した修正・失敗ログの要点をユーザーへ報告する(グリーンにできないままマージ可能な状態を装わない)。

### 8.4. 規律(厳守)

- **判定基準を緩めてグリーンにしない**: テストの削除・スキップ・アサーションの弱体化・CI 定義からのジョブ / ステップの除外で通過させない。修正の対象はコードであり、判定基準ではない。CI 定義自体の欠陥(実行環境の不一致・依存キャッシュの破損等)が根本原因の場合に限り CI 定義を修正し(type は `ci`)、根本原因の判断根拠をコミット本文に記す。
- **フレーキーテストの扱い**: 変更と無関係な失敗が再実行(`gh run rerun --failed`)だけで通った場合は、フレーキーとして記録し(Issue 起票または報告)、黙って再実行で済ませない。再実行もラウンド数に数える。
- PR の自動マージ・自己承認(approve)・強制 push を行わない。
- CI 失敗の修正でも安全制約(6.)・選択的ステージングを守る。修正コミットの type は原因に応じる(`fix` / `test` / `ci` 等)。

## 9. 注意

- 上記はバックグラウンドの定型処理であり、ワークフロー本来の停止条件(ユーザー承認待ち等)を変えない。各スキルは保存 → commit/push → 承認依頼の順で進む。
- ユーザーが明示的に「コミットしない」方針を示している場合はそれに従う。
