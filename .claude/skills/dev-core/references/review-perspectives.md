# レビュー観点カタログ

dev-reviewer(敵対的判定器)に注入する観点(ペルソナ)の正本。呼び出し側(部品・composition)は作業内容に応じて観点を選び、**1 観点 = 1 起動**で dev-reviewer を並列 fan-out し、結果を保守的にマージする(1 観点でも `[Critical]` があれば全体不合格)。例外として、生成直後の一次検証(部品内蔵の文書ゲート。[doc-gate-prompt.md](../templates/doc-gate-prompt.md))は、文書ゲート系 3 観点を 1 体に包括する軽量形を用いる(観点別の並列パネルは composition の承認ゲートが担う)。

- 観点の追加・差し替えは、利用側プロジェクトの知識 port(`name: review-perspectives`。配置は自由、規約は [ports.md](./ports.md))で行える。port に同名の観点があれば port を優先する。
- 各観点の「重点」は網羅列挙ではなく検査の起点である。判定器は敵対的既定(不合格から出発し、立証できた項目のみ合格)で臨む。

出典: 観点の体系は [addyosmani/agent-skills](https://github.com/addyosmani/agent-skills) のペルソナ(code-reviewer / test-engineer / security-auditor / web-performance-auditor)の専門手順を取り込んで構成した。

## 1. 返却契約(全観点共通の既定)

起動時プロンプトで上書きされない限り、次の構造化フィールドで返す。

- `ASPECT`: 担当観点名
- `VERDICT`: `APPROVED` | `REJECTED`(`REJECTED` は `[Critical]` が 1 件以上、または `UNVERIFIED` が `なし` 以外のときに限る)
- `FINDINGS`: 指摘の一覧。各指摘に `[Critical]`(合格を妨げる)/ `[Nit]`(軽微)/ `[FYI]`(観点外の気づき)を付け、箇所(ファイル・行・ID)と反する基準を明記する
- `UNVERIFIED`: 手段が無い・環境を起動できない等で実行できなかった検証項目の一覧。なければ `なし`。`なし` 以外のときは `APPROVED` を返さない(実行不能は不合格側に倒す)。呼び出し側はこのフィールドで欠陥(修正で解消する)と未検証(人間エスカレーションで解消する)を区別する
- `EVIDENCE`: 判定の根拠(実行したコマンド・照合した資料)
- `RATIONALE`: 判定理由の要約

## 2. コード検証系の観点

実装後のコード(タスクレビュー・最終検証・出荷可否)に適用する。

### 2.1. requirements-conformance(要件整合)

- 対象: 実装・テストと、渡された受け入れ基準([notation.md](./notation.md) の要件 ID)
- 重点: 全受け入れ基準 ID が実装とテストに反映されているか(前方トレース)。要求されていない実装(スコープ外変更・過剰実装)がないか(後方トレース)。タスク境界(`_Boundary:_`)外のファイル変更がないか

### 2.2. structure(コード構造)

- 対象: 変更されたコードの可読性・アーキテクチャ
- 重点: 命名と制御フローの明快さ。モジュール境界と依存方向の整合(内側が外側に依存していないか)。既存パターンとの一貫性。過度な抽象化・不要な間接層。コードの重複。コメントが Why not(採らなかった選択肢・コードから読めない制約)に限られているか(処理を言い換えるだけのコメントは指摘対象。[durable-info.md](./durable-info.md))。契約とドメインモデルの完全性([contract-and-domain.md](./contract-and-domain.md)): 不変条件を破れる公開状態(無条件 setter・公開ミュータブルフィールド)がないか、ドメインロジックがコントローラ・サービス層へ流出(貧血モデル化)していないか

### 2.3. security(セキュリティ)

- 対象: 変更されたコードと、その信頼境界
- 重点は 6 領域を体系的に検査する:
  1. 入力処理: インジェクション(SQL/コマンド/パス)・XSS・安全でないデシリアライズ・ファイルアップロード
  2. 認証・認可: パスワード・セッション管理・IDOR(認可漏れ)・権限昇格
  3. データ保護: 秘密情報のハードコード・ログへの漏出・暗号化・PII の扱い
  4. インフラ: セキュリティヘッダ・CORS・依存ライブラリの既知脆弱性
  5. 外部連携: Webhook 検証・SSRF・外部 API への過剰な信頼
  6. AI/LLM 機能: プロンプトインジェクション・ツールの過剰な権限
- 信頼境界ごとに STRIDE を適用し、指摘は [OWASP Top 10](https://owasp.org/www-project-top-ten/)・CWE へマッピングする。悪用可能性を検証してから報告する(理論上の指摘と実証済みの指摘を区別する)

### 2.4. performance(パフォーマンス)

- 対象: 変更されたコードの実行効率と退行
- 重点: N+1 クエリ・無制限ループ・不要な再計算/再レンダリング・メモリリーク。性能目標が定義されている場合は主要パスを計測して照合する
- 計測の誠実さ(Metric Honesty): 計測値を捏造しない。実測か静的解析かの出所を必ずラベルする。実測できない環境では「potential impact」と明示する

### 2.5. test(テスト品質)

- 対象: 変更に伴うテストコード
- 重点: 実装ではなく振る舞いをテストしているか。操作の結果(状態・出力)を検証しているか。モックはシステム境界のみに限定されているか。異常系・境界値・並行性が検討されているか。契約([contract-and-domain.md](./contract-and-domain.md))が検証されているか(事前条件違反の拒否・事後条件・不変条件)。既存の振る舞いを守る回帰テストがあるか。テスト全体がグリーンか(差分外を含む)
- バグ修正のレビューでは、修正前に失敗するテスト(再現テスト)で立証されているかを確認する(Prove-It)

### 2.6. runtime-smoke(実行時スモーク)

- 対象: 変更後のアプリケーションの実行時挙動(起動・主要フロー・GUI の描画・認証の成立)
- 追加条件: 実行時挙動に影響する変更では必須で追加する(固定観点に準ずる)。外してよいのは実行面を持たない変更(ドキュメント等)のみ
- 重点: [runtime-verification.md](./runtime-verification.md) の検証項目(共通・GUI・認証)を、同リファレンスの手段(能力検出)で実際に実行して観察する。テスト・ビルドのグリーンを実行時成立の根拠にしない。実行できない項目は返却の `UNVERIFIED` フィールドに列挙する(1. の規約により APPROVED を返せない)

## 4. 文書ゲート系の観点

中間生成物(spec.md・tasks.md 等)の承認ゲートに適用する。

### 4.1. correctness(正しさ)

- 重点: 上流(依頼・上流成果物)の意図と内容が一致しているか。事実・参照(ID・パス)が実在するか。論理矛盾がないか。体裁が整っていることを合格の根拠にしない

### 4.2. testability(検証可能性)

- 重点: 受け入れ基準・完了条件が検証可能な形(測定可能・二値判定可能)で書かれているか。曖昧語([notation.md](./notation.md) §4)が残っていないか。1 基準 = 1 検証可能な振る舞いか

### 4.3. completeness(網羅性)

- 重点: 異常系・エッジケースが検討されているか。スコープ境界(やらないこと)が明示されているか。未解決マーカー(`[要確認:]`・`UNVERIFIED`)が残っていないか。下流が作業を開始できる自己完結性があるか
