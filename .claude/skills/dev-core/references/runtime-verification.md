# 実行時検証(常設 DoD)

実行時検証(変更後のアプリケーションを実際に起動し、利用者と同じ経路で挙動を観察する検証)の正本。テスト・ビルド・リントのグリーンは実行時の成立の根拠にならない。単体テストは CSS・レイアウト・実レンダリング・Cookie の往復・セッション永続を検証せず、この種の不備はテストが全てグリーンのまま完成後に露見する。そこで「実行時に検証済みであること」を、タスク別の受け入れ基準とは別の、全変更共通の完了基準(Definition of Done)とする。

出典: [addyosmani/agent-skills](https://github.com/addyosmani/agent-skills) の definition-of-done(実行時に検証済みであること。コンパイル・型チェックだけでは足りない)と browser-testing-with-devtools(ブラウザ実行時検証の手順)。

要請レベルは RFC2119 に対応する日本語の語句(しなければならない = MUST、する必要がある = SHOULD、してもよい = MAY)で表す。

## 1. 適用範囲

- 対象: 実行時挙動に影響する変更(製品コードの追加・変更、および実行時に読まれる設定)。
- 対象外: ドキュメント・コメント・実行時に読まれない設定のみの変更。
- 適用点は 3 層とする。上の層で検証済みの項目を下の層で重複実行する必要はないが、判定の独立性(4.)は最終検証・出荷の層で必ず満たす。
  1. タスク検証(dev-implement): 実装者が、変更したフローを起動して観察する
  2. 最終検証パネル(dev-implement): 独立文脈の判定器が runtime-smoke 観点で作業単位全体を検証する
  3. 出荷ゲート(dev-release): 同上を出荷スコープ全体で行う

## 2. 検証項目

検証項目は要件(受け入れ基準)と注入知識(視覚設計・ドメインの port)から導く。次は最低限の既定であり、該当する変更で省略してはならない。

| 区分       | 項目                                                                                                                                                                                                |
| ---------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 共通       | アプリケーションが起動する。変更したフローを 1 往復して期待どおり動く。console・サーバログにエラーが 0 件                                                                                           |
| GUI を持つ | 全主要ルートで共通レイアウト(ヘッダ・ナビゲーション等)が可視。指定デザイントークンが適用されている(computed style またはスクリーンショットで確認)。素の UA スタイルのまま・未スタイルのページが無い |
| 認証を持つ | ログイン正常系が 1 往復通る(実バックエンド)。ログイン後にリロードしてもセッションが維持される。ログアウトできる                                                                                     |

認証・セッション境界の検証をモック(jsdom・偽 Cookie)で代替してはならない。モックで緑のテストは認証成立の根拠にならない(Cookie 属性・CORS・初期ユーザー不在の不備は実行時にしか現れない)。

## 3. 手段(能力検出)

特定のツールを前提にしてはならない。環境で利用可能な手段を上から順に検出して使う。

1. ブラウザ自動化(環境に構成済みの場合): ブラウザ操作ツール(DevTools・Playwright 等)。スクリーンショット取得・console 取得・computed style・ネットワーク観察
2. HTTP レベル(curl 等): 起動・ヘルスチェック・ログイン往復(Set-Cookie の属性、Cookie 付き再リクエストでのセッション維持の確認)
3. プロセスレベル: サーバ・CLI を起動してログを観察

- dev スキル群のコアは、この検証のために hooks・MCP 等の常駐機構を導入しない(依存の規律)。本節は環境に既にある手段の利用だけを指し、使える手段が無ければ 5. に従って縮退する。
- GUI の見た目の確認はブラウザ自動化(1.)がある場合に限り実測できる。無い場合、視覚項目は UNVERIFIED であり、HTTP 応答の HTML を読んだ結果で代替して合格にしてはならない。

## 4. 判定の独立性

- 最終検証・出荷の層では、実行時検証の合否は生成主体と別の独立文脈(dev-reviewer の runtime-smoke 観点)が判定しなければならない。生成と相関する盲点(自分の生成物を「スタイルが当たっている」と誤認する等)は自己判定では除去できない。
- 判定器に渡すのは成果物(変更・起動手順・検証項目)のみとし、作成者の自己評価・「動作確認済み」という主張を渡してはならない(同意への誘導になる)。
- 敵対的既定: 不合格から出発し、観察で立証できた項目のみ合格とする。

## 5. 実行不能時の扱い

- 手段が無い・環境を起動できない場合、その項目を UNVERIFIED と明示する。実行できなかった検証を合格扱いにしてはならない。
- UNVERIFIED を含む場合、最終検証・出荷ゲートは GO を出さず、未検証項目の一覧と人間が実施すべき確認手順を添えてエスカレーションする(人間の確認なしに完了・出荷にしない)。
- 「環境があれば実行する」という任意扱いを採らない。実行するか、実行できないことを明示して人間判断へ渡すかの二択とする。

## 6. 記録

- 実行した手段・コマンド・観察結果(スクリーンショットの保存先・console 出力の要点・HTTP 応答の要点)を、EVIDENCE(レビュー返却)または Implementation Notes に記録する。
- 出所をラベルする: 実測(観察した)/ 静的(コードから推定した)/ UNVERIFIED(観察していない)。観察していない項目を実測のように書いてはならない。

## 7. 自己正当化への反駁

検証を省略する典型的な言い分と、その反駁。省略を判断する前に照合する。

| 言い分                             | 反駁                                                                                                                           |
| ---------------------------------- | ------------------------------------------------------------------------------------------------------------------------------ |
| テストが通るなら画面も正しいはず   | 単体テストは CSS・レイアウト・実レンダリングを検証しない。トークン未適用・ヘッダ欠落はテストのグリーンに現れない               |
| 認証テストが緑だからログインできる | モック・jsdom の緑は実ブラウザ + 実バックエンドの成立根拠にならない。Cookie 属性・CORS・初期ユーザー不在は実行時にしか現れない |
| コードを読んだ限り正しい           | 実行時挙動はコードの印象と定常的に食い違う。読解は観察の代替にならない                                                         |
| 環境が無いので飛ばす               | 飛ばしてよい。ただし UNVERIFIED と明示して人間へ渡す。黙って合格にしない                                                       |
| 起動して見たが問題なさそう         | 「なさそう」は根拠にならない。検証項目と観察結果を 1 対 1 で対応させる                                                         |
