# 契約による設計とドメインモデルの完全性

仕様(dev-spec)と実装(dev-implement)が共有する設計・実装規範の正本。レビュー(観点カタログの structure / test 観点)はここに照らして検査する。プロジェクト固有の追加・上書きは知識 port(`name: principles`。規約は [ports.md](./ports.md))で行える(port 優先)。

## 1. 契約による設計(Design by Contract)

コンポーネント間のインターフェースを「契約」として明示する(出典: [Bertrand Meyer, Design by Contract(Eiffel Software)](https://www.eiffel.com/values/design-by-contract/introduction/))。

| 契約                    | 意味                                       | 責務                                                                   |
| ----------------------- | ------------------------------------------ | ---------------------------------------------------------------------- |
| 事前条件(precondition)  | 呼び出し側が満たすべき条件(引数・状態)     | 呼び出し側が保証する。実装はガード節・型で検査し、違反は即座に拒否する |
| 事後条件(postcondition) | 処理完了時に成り立つ結果の保証             | 実装側が保証する。テストの検証対象になる                               |
| 不変条件(invariant)     | オブジェクトの生存期間中、常に成り立つ条件 | オブジェクト自身が保証する(2. の完全性で実現する)                      |

- **設計での宣言**: インターフェース定義(spec.md の公開インターフェース・データ構造)に事前条件・事後条件・不変条件を明記する。暗黙の前提を契約の外に残さない。
- **実装での強制**: 事前条件はガード節と型で表現する(可能な限り、違反を型システムで表現不能にする)。事後条件・不変条件はテストで検証する。
- **永続化との関係**: 契約はテストコード(What)として永続化する([durable-info.md](./durable-info.md))。契約の理由・破ると壊れる前提は doc コメント(Why not)に書く。

## 2. ドメインモデルの完全性

### 2.1. 不変条件の常時充足(always-valid)

不正な状態のドメインオブジェクトを**存在させない**(出典: [Vladimir Khorikov, Always-valid domain model](https://enterprisecraftsmanship.com/posts/always-valid-domain-model/))。

- **完全コンストラクタ**: 生成時点で不変条件をすべて検査し、満たさない生成を失敗させる。生成後に必須項目を埋める「不完全な生成 + 後付け設定」を許さない。
- **更新も不変条件を通す**: 状態変更は意図を表すメソッド経由に限定し、そこで不変条件を再検査する。無条件の setter・公開ミュータブルフィールドを設けない。
- **表現不能化を優先**: 検査(実行時バリデーション)より、型で不正状態を表現できなくする設計を優先する(例: null 許容をやめる、値オブジェクトで制約を型に埋め込む)。
- 帰結: ドメインオブジェクトを受け取った側は、その妥当性を再検査しなくてよい(検証の重複と「どこで検証済みか分からない」状態を排除する)。

### 2.2. ドメインロジックの集約

ドメインのルール・計算・判断はドメイン層(ドメインオブジェクト)に集約する。データ保持だけのドメインオブジェクトとロジックを持つサービス層の組(貧血ドメインモデル)を避ける(出典: [Martin Fowler, AnemicDomainModel](https://martinfowler.com/bliki/AnemicDomainModel.html))。

- 同じドメインルールが複数箇所(コントローラ・サービス・UI)に重複したら、ドメイン層への集約対象とみなす。
- 完全性(すべてのロジックをドメイン層に)と純粋性(ドメイン層からプロセス外依存を排除する)が衝突する場合は、純粋性を優先しドメイン層を副作用から隔離する。その判断は仕様文書 spec.md §8 実現方針に記録する(出典: [Vladimir Khorikov, Domain model purity vs. domain model completeness](https://enterprisecraftsmanship.com/posts/domain-model-purity-completeness/))。

## 3. 各工程での適用

| 工程                   | 適用                                                                                                                     |
| ---------------------- | ------------------------------------------------------------------------------------------------------------------------ |
| 仕様(dev-spec)         | 公開インターフェース・データ構造の契約に事前条件・事後条件・不変条件を明記し、ドメインルールの集約先を示す |
| 実装(dev-implement)    | ガード節・型・完全コンストラクタで契約を強制。契約(事後条件・不変条件・事前条件違反の拒否)をテストで検証                 |
| レビュー(観点カタログ) | structure: 不変条件を破れる公開状態・ロジックの流出の検出。test: 契約の境界(事前条件違反・不変条件)の検証有無            |
