# ソース駆動の根拠提示(Detect → Fetch → Cite)

フレームワーク/ライブラリ固有の API・設定・コードを設計・実装に用いるとき、**記憶や推測ではなく、使用バージョンの公式ドキュメントを根拠**にするためのプロトコル。設計・実装を行う部品が参照する。本スキル群の「事実には根拠(出典)を常に示す」という原則(根拠主義)を、外部依存のコードへ適用するもの。

標準規格(ISO/OWASP 等)の根拠は知識 port(標準カタログ)が扱うのに対し、本リファレンスは **個別の依存ライブラリの当該バージョン公式情報** を扱う。両者は補完関係。

## 1. なぜ必要か

- LLM の記憶は学習時点で固定され、ライブラリの API は版ごとに変わる。記憶ベースの実装は **陳腐化した/存在しない API** を生みやすい。
- 「動くはず」の推測は、バージョン不一致・破壊的変更・非推奨化で静かに壊れる。出典を辿れる根拠に置き換える。

## 2. プロトコル

### 2.1. Detect(バージョンの特定)

コードを書く前に、対象ライブラリの**実際に使われているバージョン**を依存定義から特定する。

- 例: `package.json`/`package-lock.json`・`requirements.txt`/`poetry.lock`・`go.mod`・`Cargo.toml`・`*.csproj`・`pom.xml`/`build.gradle` 等。
- ロックファイルがあればロックファイルの解決済みバージョンを正とする。

### 2.2. Fetch(当該バージョンの公式情報を参照)

特定したバージョンに対応する**公式の該当ページ**を参照する(トップページではなく、使う API の具体ページ)。

- 根拠の権威階層(高い順): (1) 当該版の公式ドキュメント/API リファレンス → (2) 公式リポジトリのソース・型定義・CHANGELOG・リリースノート → (3) 信頼できる二次情報(メンテナの記事等) → (4) 一般のブログ・Q&A(最終手段、要裏取り)。
- メジャー版が異なる情報は、破壊的変更の可能性があるため当該版で裏取りする。

### 2.3. Cite(引用)

設計・実装の根拠として、参照した**完全な URL(可能ならアンカー付きの該当箇所)**を残す。

- 設計を行う部品は設計文書の技術選定の理由欄や調査メモに、実装を行う部品はサブエージェントの返却(NOTES 等)や PR/コミット本文に出典を残す。

### 2.4. UNVERIFIED(検証不能の明示)

公式情報で確認できなかった事項は、**推測で確定させず `UNVERIFIED` と明示**してエスカレーションする。

- 例: 「この API の引数順は `UNVERIFIED`(vX.Y のドキュメントに記載が見当たらず、要確認)」。
- 記憶だけを根拠に未確認の API を断定実装しない。確認できないなら、確認できる代替手段(型定義の参照・最小実行確認)を取るか、上流へ差し戻す。

## 3. 適用の目安

- **適用する**: 外部フレームワーク/ライブラリ固有の API・設定・規約を用いるコード、バージョン依存の挙動。
- **過剰適用しない**: 言語標準・一般的アルゴリズム・プロジェクト内コードなど、バージョン依存でなく自明なものに逐一出典を求めない(JBGE)。

## 4. 出典(手法の背景)

- セマンティックバージョニングによる破壊的変更の扱い: https://semver.org/
- 一般に、依存の API は版ごとに変わりうるため、当該版の一次情報を根拠とする。
