# 恒久情報の配置規約

「コードが SSoT」([principles.md](./principles.md) §1)の下で、コードだけでは表現しにくい恒久情報をどこに置くかを定める。ここに挙げる置き場以外の恒久文書を作ってはならない。

中核の規範は次の 4 行である(出典: [和田卓人氏による定式化](https://x.com/t_wada/status/904916106153828352))。

```txt
コードには How
テストコードには What
コミットログには Why
コードコメントには Why not
```

## 1. 配置先の一覧

| 情報                                                              | 置き場                         | 規約                                                                                                                                                              |
| ----------------------------------------------------------------- | ------------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| How(実装がどう動くか)                                             | コードそのもの                 | コード自体に語らせる。How をコメントで言い換えない                                                                                                                |
| What(何をするべきか = 振る舞いの仕様・受け入れ基準)               | テストコード                   | 実行可能な仕様として永続化する(2. を参照)                                                                                                                         |
| Why(なぜこの変更をしたか)                                         | コミットログ                   | 本文に変更理由・経緯を書く([git-convention.md](./git-convention.md))                                                                                              |
| Why(なぜこの構造・技術か = 恒久的に参照するアーキテクチャ決定)    | ADR: `docs/adr/NNNN-<slug>.md` | Nygard 形式(Title / Status / Context / Decision / Consequences)。1 決定 1 ファイル、連番。コミットログの Why のうち、変更単位を超えて参照され続けるものを昇格する |
| Why not(なぜ他のやり方をしなかったか・コードから読み取れない制約) | コードの doc コメント          | 採らなかった選択肢とその理由、破ると壊れる前提のみ書く。処理の言い換え(How/What)や中間生成物の ID(要件 ID・タスク番号)を書かない(4. を参照)                     |
| 運用・横断方針                                                    | README(対象ディレクトリに同居) | セットアップ・運用手順・横断的な方針など、コード単位に紐づかない最小限の記述                                                                                      |

出典: [Michael Nygard, Documenting Architecture Decisions](https://cognitect.com/blog/2011/11/15/documenting-architecture-decisions)。

## 2. 受け入れ基準のテストへの引き継ぎ

- 中間生成物 spec.md の受け入れ基準は、実装時にテストコードへ引き継ぐ。テスト名・説明は検証する振る舞いが分かる自然言語で書く。
- 実装完了(凍結)後の受け入れ基準の正はテストコードである。凍結された spec.md は参照用の履歴にすぎない。
- 既存の振る舞いを守る必要が生じたら、その振る舞いにテストを付けてから変更する(振る舞いの契約化)。

## 3. 反映の規律

- 実装後に判明した仕様との差異・新たな決定は、凍結された中間生成物ではなく本書の配置先(テスト・doc コメント・ADR・README)へ反映する。
- 中間生成物に恒久情報を書かない。中間生成物は「実装までの作業指示」であり、実装完了で役目を終える。
- 納品物として恒久的な自然言語ドキュメントが必要な場合は拡張(Layer 3)が生成する。それは成果物であって SSoT ではないため、本規約の対象外。

## 4. トレーサビリティをコードに持ち込まない

- コードに中間生成物の ID を残さない。要件 ID・タスク番号を、コードコメント・識別子・文字列リテラルへ書かない。
- 理由: 中間生成物は実装完了で凍結され参照専用になる([principles.md](./principles.md) §1)。コードに ID を残すと、コードから凍結文書への参照が生まれ、以後の仕様変更(新しい作業単位での再実装)で参照先が古くなりドリフトの起点になる。凍結された中間生成物にはコードとの整合義務がないため、この参照は保守されない。
- トレーサビリティは中間生成物の内部(requirements → design → tasks)で閉じる([notation.md](./notation.md))。コードは中間生成物を後方参照しない。実装が満たす受け入れ基準は、2. のとおりテストコードへ引き継ぐ(トレース先はテスト名・アサーションが表す)。
