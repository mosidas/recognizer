# 記法規約

中間生成物(spec.md・tasks.md 等)で部品間が参照し合う ID・注記・マーカーの記法を定める。部品はこの記法で読み書きすることで、他の部品の存在を知らずに連携する(ファイル契約)。

## 1. 要件 ID

- 要件は数値 ID で識別する。要件単位は `Requirement 1:` 形式、受け入れ基準は `1.1`、`1.2` の階層番号を付ける。
- ID は昇順の連番とし、欠番・重複を作らない。
- 下流の成果物(tasks)から上流の要件を参照するときは、この ID を使う。実在しない ID を参照してはならない(捏造禁止)。

## 2. トレーサビリティ注記

タスクには、次のイタリック注記を付ける。

| 注記                        | 意味                                                                                                           | 例                                   |
| --------------------------- | -------------------------------------------------------------------------------------------------------------- | ------------------------------------ |
| `_Requirements: <ID 列>_`   | 対応する受け入れ基準の ID                                                                                      | `_Requirements: 1.2, 2.1_`           |
| `_Boundary: <名前>_`        | 責務を持つコンポーネント境界                                                                                   | `_Boundary: AuthService_`            |
| `_Depends: <タスク番号列>_` | 依存する先行タスク                                                                                             | `_Depends: 2, 3_`                    |
| `(P)`                       | 境界が独立しており並行実行可                                                                                   | `- [ ] 4. (P) ...`                   |
| `_Knowledge: <name 列>_`    | 知識 port の明示上書き(通常は書かない。選択は port マッピングに任せ、特定 port の注入を固定したいときのみ書く) | `_Knowledge: lang-typescript, auth_` |
| `_Blocked: <根本原因>_`     | 有界リトライで解決しなかったタスクの印                                                                         | `_Blocked: 外部 API の仕様が未確定_` |

## 3. マーカー

| マーカー           | 意味                                   | 解消方法                                                         |
| ------------------ | -------------------------------------- | ---------------------------------------------------------------- |
| `[要確認: <論点>]` | 値・方針が未確定。推測で埋めていない印 | ユーザーへの質問で確定し、マーカーを除去する                     |
| `UNVERIFIED`       | 出典で確認できていない事実             | 公式情報の確認([source-driven.md](./source-driven.md))で解消する |

- マーカーを残したまま承認ゲートを通過してはならない(残存はチェッカ・レビューの検出対象)。

## 4. 曖昧語の禁止と定量化

- 「適切に」「高速に」「十分な」「柔軟に」等の曖昧語で要求・基準を書いてはならない。
- 測定可能な形で書く: メトリクス(単位)・目標値または閾値・前提条件・測定方法。
- 例: 「高速に応答する」ではなく「一覧 API の応答時間が p95 で 300 ms 以下(同時 100 接続、ステージング環境で計測)」とする。

## 5. 文書の書式

- 中間生成物は Markdown(GFM)で記述する。
- 見出し構成・セクションの並びは各部品の `templates/` が正本であり、本書では定めない。
