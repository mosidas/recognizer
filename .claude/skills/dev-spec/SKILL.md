# dev-spec — 壁打ちによる仕様(契約 + 受け入れ基準)の生成

依頼内容を対話で「外から見える契約(公開インターフェース・データ構造)」として確定し、その振る舞いを EARS 受け入れ基準に落として `<workdir>/spec.md` を生成するサブ部品。requirements.md / design.md の 2 文書構成の統合ではなく、壁打ちで決まる契約を中心に据えた 1 文書構成をとる。出力は実装までの中間生成物であり、実装完了で凍結される(受け入れ基準はテストコードへ引き継がれる)。

## 1. 契約

- **workdir**: 呼び出し時の引数で指定する。既定は `docs/dev/`(composition は上書きする)。
- **入力**: 依頼文(引数または対話)。既存の `<workdir>/spec.md` があれば更新モード(全置換せず差分更新)。
- **出力**: `<workdir>/spec.md`(公開インターフェース・データ構造・EARS 受け入れ基準・要件 ID 連番)。調査ログは `<workdir>/research.md`(任意)。
- **port**: 差し替え port `hearing-questions`(name 参照。ヒアリング質問セット)。知識 port を含む選択はポートマッピング(`../dev-core/references/ports.md`)に従う(inject に dev-spec を含むもの。例: 原則との整合、機能ドメインの知識)。

## 2. 参照(必読)

- EARS 記法: `../dev-core/references/ears-format.md`
- 記法規約(要件 ID・マーカー・曖昧語禁止): `../dev-core/references/notation.md`
- 契約による設計・ドメインモデルの完全性: `../dev-core/references/contract-and-domain.md`
- ソース駆動の根拠提示: `../dev-core/references/source-driven.md`(外部ライブラリの API を契約に用いるとき)
- Git 運用規約: `../dev-core/references/git-convention.md`
- 文書ゲート: `../dev-core/templates/doc-gate-prompt.md`
- テンプレート: `./templates/spec-template.md`

## 3. ステップ

### Step 1: コンテキスト読込

- 依頼文と、既存の `<workdir>/spec.md`(あれば)を読む。
- 知識 port をポートマッピング(`../dev-core/references/ports.md`。走査は `ports.py --skill dev-spec`)に従って読み、原則・規律と矛盾しない仕様にする。差し替え port `hearing-questions` の有無もこの走査結果で確認する。
- **既存コントラクトの確認(既存コードベース拡張時)**: エラー型とステータスコード・API クライアントの分岐・既存の公開インターフェース・採用済みテスト基盤など、既存実装の実シグネチャを確認し、推測で仕様化しない。大量の読み込みが要る場合は dev-explorer に隔離してダイジェストのみ受け取り、調査ログを `research.md` に残す。
- workdir が凍結済み(state.json が完了状態)の場合は再生成せず停止し、新しい作業単位(別 workdir)を案内する。

### Step 2: 目的・スコープ・前提の確定(質問駆動)

壁打ちに入る前に、曖昧な点を `AskUserQuestion` で確認する(1〜3 問ずつ)。

- **既定(簡易)**: 推測禁止項目 = 利用者・ロール / 準拠レベル(セキュリティ・規制) / 技術スタック・インフラ制約 / ドメインルールのうち、依頼文から確定できないものだけを質問する。加えて、機能の境界(含む/含まない)と非機能要件の有無を確認する。
- **GUI を持つ作業の必須確認**: 事前に指定された視覚設計の正本(デザイントークン定義・共通レイアウト仕様・デザインシステム・UI モック)の有無と所在を必ず確認する。
- **認証を持つ作業の必須確認**: 認証方式(セッション Cookie・トークン等)、初期ユーザー(seed)の供給方法、セッション永続戦略(リロード・再訪問後の復元経路)、セッション失効時の想定挙動のうち、依頼文から確定できないものを必ず質問する。
- **port 差し替え時**: `name: hearing-questions` の port があれば、その質問セットに従う。
- ユーザーが「お任せ」と答えた項目は、選んだ案と理由を spec.md の「前提」に明記する。

### Step 3: 契約の壁打ち(公開インターフェース・データ構造)

この部品の中心工程。契約のドラフトを提示し、対話で往復して確定する。

- **ドラフト提示**: §5(公開インターフェース)と §6(データ構造)の具体案(シグネチャ・型定義の擬似コード)を先に書いて見せる。文章での説明より先に契約の形を出し、ユーザーが差分で意見できるようにする。
- **契約による設計**(`../dev-core/references/contract-and-domain.md`): 各インターフェースに事前条件・事後条件を明記する。データ構造には不変条件を列挙し、生成時に強制する設計(完全コンストラクタ・型による表現不能化)を示す。ロジックの集約先を明示する(貧血ドメインモデルを避ける)。
- **エラーの表出**: 異常系の分類と呼び出し側への表出のしかたを契約に含める。既存のエラー型があれば実コードを確認して整合させる。
- **脅威モデル(セキュリティ関連時)**: 認証・認可・外部入力・機微データを扱う契約では、信頼境界を明示し、STRIDE で脅威を洗い出して契約とエラー表出・受け入れ基準(異常系)に反映する。
- **実装判断の記録**: 壁打ちで確定した実装上の判断(技術選定と理由・既存構造との関係)は §8「実現方針」に要点のみ記す。内部のファイル構成・実装手順は書かない(dev-decompose の責務)。
- 往復はユーザーが契約に合意するまで続ける。未確定論点は `[要確認: <論点>]` マーカーで明示し、推測で埋めない。

### Step 4: 受け入れ基準の生成(EARS)

合意した契約の振る舞いを `./templates/spec-template.md` §7 の形式で書き下ろす。

- 各要件は数値 ID(`Requirement 1:`)を持ち、「対象」欄で §5/§6 の要素(インターフェース名・型名)を指す。契約に紐付かない振る舞いを書かない(紐付け先が無いなら §5–6 の欠落として Step 3 に戻る)。
- 受け入れ基準は EARS の 5 パターンで記述し、`1.1`, `1.2` と番号付けする。異常系・エッジケースを必ず検討する。セキュリティ関連機能では abuse case(不正入力・権限逸脱・なりすまし等)を異常系受け入れ基準の発生源にする。
- 認証を持つ機能では、正常系の成立と永続性を必ず含める: ログインできること(イベント)、リロード・再訪問後もセッションが維持されること(状態)、失効時の挙動(異常系)。
- GUI を持つ機能では、共通レイアウトの適用・指定デザイントークンへの準拠を受け入れ基準化する。
- データ削除のカスケードで新たに生じる状態(null 化した参照・親を失ったレコード等)の操作を状態系/異常系として定義する。
- 非機能要件も同じ形式で書き、曖昧語を使わず定量化する(`../dev-core/references/notation.md` §4)。

### Step 5: 内蔵レビューゲート

`../dev-core/templates/doc-gate-prompt.md` で dev-reviewer を独立文脈で 1 体起動し、文書ゲート系 3 観点(correctness / testability / completeness)で判定させる。種別固有チェックとして以下を渡す。

- 網羅性: 依頼・壁打ちで確定したスコープを漏れなくカバーしているか。任意とされた項目を勝手に必須化していないか。
- 契約の完全性: 全公開インターフェースに事前条件・事後条件・エラー表出があるか。データ構造に不変条件とロジックの所在があるか。
- 紐付けの完全性: 全 Requirement の「対象」が §5/§6 に実在するか。逆に、どの Requirement からも参照されない §5/§6 の要素(振る舞い未定義の契約)がないか。
- EARS 準拠: 各受け入れ基準が EARS 構文と数値 ID 規則に従うか。
- スコープ境界: やらないことが理由付きで明示されているか。前提が検証方法付きで書かれているか。
- 機能横断の必須基準: 認証(ログイン正常系・セッション永続・失効時挙動)・GUI(共通レイアウト・デザイントークン準拠)の受け入れ基準があるか。
- 参照の実在: 引用した既存コード・文書・外部 API の参照が実在するか(捏造禁止)。

`REJECTED` なら指摘を反映して**最大 2 回**自己修復する。`QUESTIONS` があれば `AskUserQuestion` でユーザーに確認して解消する(推測で埋めない)。

### Step 6: 保存とコミット

- `<workdir>/spec.md`(必要に応じ `research.md`)を保存する。
- `check.py --workdir <workdir>` で機械検査する(要件 ID の連番・残存マーカー・曖昧語。state.json がある場合は `--def` も付けて状態検査。詳細は `../dev-core/references/static-check.md`)。warning は内容を確認して Step 5 の自己修復に戻す。
- `../dev-core/references/git-convention.md` に従い選択的にステージして commit & push する(例: `docs(<unit>): 仕様(契約と受け入れ基準)を生成`)。

### Step 7: 停止

- spec.md を提示し、ユーザーのレビューを待って停止する。**状態遷移(承認)はこの部品では行わない**(composition またはユーザーが状態機械を操作する)。
- 次の部品として dev-decompose(タスク分解)を案内する。

## 4. 注意

- spec.md は「何を・外から見てどう振る舞うか」の文書。内部実装の詳細(ファイル構成・実装手順)を書かない。File Structure Plan は dev-decompose の責務。
- 要件番号は tasks からトレースされる。安易な番号変更を避ける。
- 更新モードでは既存の要件 ID を保持し、追加は末尾連番、削除は欠番として理由を残す。
- 本部品は spec.md 1 本に一本化する。要件・設計を別文書(requirements.md + design.md)に分ける 2 文書構成は廃止した。契約(§5・§6)と受け入れ基準(§7)を spec.md の中で完結させる。
