## 記述ルール(要約)

- 要件見出しは数値 ID のみ(`Requirement 1`)。受け入れ基準は `<要件番号>.<連番>`。この ID は tasks.md の `_Requirements:` から参照される。
- 各 Requirement の「対象」欄で §5(公開インターフェース)/ §6(データ構造)の要素を指す。契約に紐付かない振る舞いを書かない(紐付け先が無いなら §5–6 の欠落を疑う)。
- 受け入れ基準は EARS の 5 パターンで、1 文 1 振る舞い、測定可能に書く。異常系・エッジケースを必ず含める。
- 認証を持つ機能は、ログイン正常系・リロード後のセッション維持・セッション失効時の挙動を受け入れ基準に含める。
- GUI を持つ機能は、共通レイアウトの適用・指定デザイントークンへの準拠を受け入れ基準に含める(視覚設計の正本があれば所在を参照で示す)。
- File Structure Plan(どのファイルに何を置くか)は書かない(dev-decompose の責務)。
- 不明点は推測せず、生成フェーズでその場で全て質問して解消する。
