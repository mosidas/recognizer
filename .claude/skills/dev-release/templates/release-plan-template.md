# リリース計画テンプレート

dev-release が出荷の実行計画を `<workdir>/release-plan.md` に書くためのひな形。本ファイルはリリースごとの一時的な計画であり、恒久的な運用情報(Runbook 相当: 運用手順・監視方針)はここに書き溜めず、README 等の恒久情報への反映を提案する(`../../dev-core/references/durable-info.md`)。
段階・判定基準・復旧時間の既定値の出典: [addyosmani/agent-skills](https://github.com/addyosmani/agent-skills) shipping-and-launch。プロジェクトの実態に合わせて調整してよい(調整した場合は理由を書く)。

```markdown
# リリース計画 — <version>

- 生成日: <YYYY-MM-DD>
- 対象 unit: <unit, ...>(すべて完了状態)
- 出荷可否: <GO / NO-GO>(根拠: release-report.md)

## 出荷前チェックリスト

- [ ] 全対象 unit の state.json が完了状態(凍結済み)
- [ ] テスト・ビルド・リント全グリーン(差分外含む)
- [ ] 機械確認チェックリスト通過(秘密情報・デバッグ残置・CORS・未確定文字列・ヘルスチェック。対象外は理由付き)
- [ ] 出荷可否パネル GO(security / performance / test + 特性の観点)
- [ ] 監視・アラートが本変更の失敗を捕捉できる

## フィーチャーフラグ

| フラグ | 所有者 | 既定値(安全側) | 段階的有効化                | 失効日・撤去条件                                |
| ------ | ------ | -------------- | --------------------------- | ----------------------------------------------- |
| <flag> | <名前> | off            | チーム/ベータ→カナリア→全体 | <日付>。全展開後 2 週間以内に削除(負債にしない) |

- フラグ OFF でデプロイしてから有効化する。フラグのネストを作らない。CI では ON / OFF 両状態をテストする。

## 段階ロールアウト

| 段階          | 対象範囲   | 滞留時間(目安)           | 監視指標(SLI 等)         |
| ------------- | ---------- | ------------------------ | ------------------------ |
| チーム/ベータ | 内部利用者 | 24 時間                  | <エラー率・レイテンシ等> |
| カナリア      | 5%         | 24〜48 時間              | 同上                     |
| 段階拡大      | 25% → 50%  | 各 24 時間               | 同上                     |
| 全体          | 100%       | 1 週間監視後にフラグ撤去 | 同上                     |

### 進行判定の基準(各段で適用)

| 指標                     | 進行                   | 保留(原因調査)         | 即時ロールバック |
| ------------------------ | ---------------------- | ---------------------- | ---------------- |
| エラー率                 | ベースライン ±10% 以内 | +10〜100%              | 2 倍超           |
| P95 レイテンシ           | ベースライン ±20% 以内 | +20〜50%               | +50% 超          |
| クライアント実行時エラー | 新種なし               | セッション比 0.1% 未満 | 0.1% 以上        |
| ビジネス指標             | 中立〜正               | 5% 未満の低下          | 5% 以上の低下    |

- 上記に加え、データ整合性の問題・セキュリティ脆弱性の発見・ユーザー報告の急増は段階を問わず即時ロールバックする。

## ロールバック手順

- トリガ条件: <上記の即時ロールバック基準 + プロジェクト固有条件>
- 戻し方(第一手段はフラグ無効化): <フラグ無効化 / デプロイ切り戻し>(各段から)。
- 復旧目標時間: フラグ無効化 1 分以内 / 前バージョン再デプロイ 5 分以内 / データベースロールバック 15 分以内(超える見込みの操作は計画段階で分離する)。
- 不可逆操作(破壊的マイグレーション・データ削除): <分離し、別承認とする。戻せないため事前バックアップの手段を書く>。
```
