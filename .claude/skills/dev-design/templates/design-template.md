````markdown
| 要件 ID | 要件内容(requirements.md より転記) | 設計要素(コンポーネント/ファイル) | 根拠・備考 |

## 7. データモデル

[ドメインモデル、データ構造、永続化スキーマなど]

- **ロジックの所在**: ドメインルール・計算・判断の集約先を明示する(貧血ドメインモデルを避ける)。

## 8. エラーハンドリング

[異常系・エラー分類・リカバリ方針。要件の異常系受け入れ基準に対応させる]

- 各異常系の分類・ステータスは既存エラー型の実コードを確認して決める。意味が合わなければ専用の型を新設する。

## 9. テスト戦略

[単体/結合/E2E の方針、テスト対象、TDD の適用範囲]

- モックはシステム境界(外部 API・非決定性)のみに限定する。時刻依存テストは時刻を固定して決定論化する。

## 10. その他(任意)

[セキュリティ / パフォーマンス / 移行 など]

- セキュリティ関連時: 信頼境界と STRIDE による脅威・対策、abuse case を記す。
- 公開 API を持つ場合: 機械可読の契約(OpenAPI 等)を正本とし、実装後の契約 diff 確認を設計に含める。

## 11. 参考資料

[research.md へのリンク、参照ドキュメント、外部 API の出典]
````

## 記述ルール(要約)

- すべての要件 ID をトレーサビリティ表でカバーする。漏れは禁止。
- File Structure Plan のパスは、プロジェクト既存の構成規約に従う。
- 知識 port の原則(あれば)に反しない(型安全性、抽象化の節度など)。
